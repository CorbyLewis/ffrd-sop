    Job Aid 15¶

    Verification of Cloud Compute Event Results¶

    Status of This Document¶

    This document is provided as a raw draft for early delivery, but has not undergone complete content review, editing or finalization.

    The ability to interact with model results that are produced via cloud compute is central to various phases of the FFRD workflow. Quality control checks to ensure that the cloud-compute generated model results match results that would be obtained via a desktop computation process is an important activity. Workflows for performing these activities within the Hydraulic HEC-RAS model are provided below.

    Workflow for Producing a Synthetic Event on Desktop (HEC-RAS)¶

    This workflow allows for the simulation of a given synthetic event using HEC-RAS on a desktop environment. It utilized cloud simulated hydrology (HEC-HMS) and reservoir operations (HEC-ResSim) outputs and therefore these datasets must be available in the Model Library.

    Note that by default, gridded excess precipitation from the HMS model is written as a HDF file. This format can be ingested by HEC-RAS as a precipitation boundary condition in a cloud environment, but not in a desktop environment. Therefore, an additional post-processing step is required in the cloud environment to generate gridded excess precipitation in HEC DSS file format, which can be ingested into RAS desktop.

    Downloading Data and File Management¶

    1. Copy full original desktop model that was used for cloud delivery. This includes the Hydraulics, Hydrology and Reservoir Operations folders. Download and verify references to external data sources (DEM, Land Cover, ResSim, HMS Baseflows, HMS Excess Precip). By downloading the hydraulic model which was executed for cloud simulation, it ensures all computational settings (such as timestep, equation set, etc.) are consistent between cloud and desktop simulations.

    2. Make a Copy of the RAS model-desktop into a new directory name, "event 2" or similar. In general, it is beneficial to mirror the directory structure of the Model Library on the desktop environment to maintain relative file paths and aid in inter-model connections.

    3. Download the Hydrology, Reservoir Simulation, and System Response directories from the SST event of interest from the computable repository, and save to the new directory (i.e. "\event 2").

    4. If the model-unit of interest requires input from an upstream model-unit, download the Hydraulics data output from the upstream model unit. By default, this output will be stored within the us-model-unit.p01.hdf file.

    Verifying Connections and Simulation Settings in the Hydraulic Model¶

    1. It can be beneficial to create a new RAS unsteady flow file and plan for the cloud event to be simulated, such as “event 7703”.

    2. Within HEC-RAS software, update the simulation window dates and times for the plan to align with SST event. This time window can be extracted from the ResSim simulation.dss timeseries data, among other locations. (Note the HMS time window for pilots has been extended 1 day prior to the rainfall to support a ResSim requirement for a “lookback period”, therefore the HMS simulation time may not be appropriate for use in RAS.)

    3. Ensure the hydraulic model is referencing the correct locations of HMS and ResSim Output. This includes: Baseflow hydrograph timeseries (from HMS), Gridded Excess Precipitation (from HMS), and Reservoir Releases (from ResSim). For timeseries boundary conditions, it can be beneficial to use the “Plot all the flow boundary condition hydrographs” button to quickly identify linkage issues with DSS connections. Also note that the simulation window must be updated prior to connecting DSS files for the plotting to succeed.

    4. Initial reservoir elevations which are sampled stochastically are adopted in the hydraulic model from either the ResSim or HMS model. In general, if a reservoir exists in the ResSim model, the initial elevation should be adopted from reservoir operations model. If it does not exist in ResSim, but does exist in HMS, it should be adopted from the hydrology model. Not all initial reservoir elevations are sampled stochastically, and therefore some may remain unchanged in the RAS unsteady flow file if they do not exist in either the ResSim or HMS models. These initial elevations can be manually extracted from either the HMS or ResSim DSS output files and should be the reservoir elevation from the initial timestep of the simulation. These initial elevations are specified for reservoir Initial Conditions (IC) Points.

      1. A secondary method to extract initial reservoir elevations is to open the model-unit.p01.hdf file for the cloud simulated RAS model for the event of interest.

        This hdf file is located in the Model Library here:

        ffrd-basin > conformance (or production) > simulations > event-data > event# > hydraulics > model-unit > model-unit.p01.hdf

        Within the model-unit.p01.hdf file, the initial reservoir elevations are specified in:

        Event Conditions > Unsteady > Initial Conditions > “IC Point Elevations” table

        (IC point names are stored in “IC Point Names” table)

    5. If the model-unit of interest requires input from one or more upstream hydraulic models, the timeseries of flows from those upstream units will need to be extracted from the upstream model units’ us-model-unit.p01.hdf file. This process can be done through a manual copy/paste exercise. The outflows from the upstream model unit will be entered into the RAS unsteady flow file as a timeseries flow hydrograph.

    Setting Up Breach Initiation Triggers¶

    1. Breach initiation is triggered at a specified water surface elevation which is sampled from the System Response Probability (SRP) curves stochastically. These breach trigger elevations must be set for each breachable structure in the RAS unsteady flow analysis window. Breach trigger elevations for individual events are stored in the failure_elevations.json file located:

      ffrd-basin > conformance (or production) > simulations > event-data > event# > system-response > failure_elevations.json

    Simulating Event and Compare Results with Cloud Simulation¶

    1. Compute hydraulic model on desktop.

    2. Compare results to ensure alignment. Note that a cloud simulated model-unit.p01.hdf result file from a given event can be downloaded and opened in HEC-RAS Mapper by dragging/dropping the file into the Mapper window. One opened, this file can be directly compared against a desktop simulated event in Mapper. However note the cloud simulated event may not display the correct simulation time window.

      Timeseries plots from a cloud simulated event can be viewed by opening the model-unit.p01.hdf file in RAS Mapper, expanding the Results > sst > Geometry component for the cloud simulated plan, then enabling a model element (such as a reference line or SA/2D connection), right clicking the element, and selecting Results Time Series > Hydrographs.

    Viewing Mapped summary Results¶

    1. Go to computable watershed in model library (e.g. FFRD Kanawha Computable).

    1. Go to the aep-grids directory.

    2. Download aep_1000.zip file.

    3. “extract to here”.

    4. Open project in QGIS.

    5. View results, if something looks fishy, you can add depth grids.

    6. Go to the runs directory, select the event number of interest.

    7. Go to the depth-grids directory.

    8. Go to the ras model of interest.

    9. Download the ras model.

    10. “extract to here” browse to the vrt file.

    11. Load the vrt file.

    12. Right click on properties, symbology load from file. Load the depth grid qml file for symbology.

    Finding a Big Event in a Location¶

    1. Go to the computable watershed in model library.

    1. Go to the statistics directory.

    2. Download the peaks.csv file.

    3. Open in excel.

    4. Find a location.

    5. Find a large value in that location.

    6. Identify the event number for that large value.

    7. Download that event number data for that location.

      Alternatively, graphical results plots can be very useful in identifying specific large events. These plots are located:

      ffrd-basin > conformance (or production) > simulations > summary-data > plots

    Verify Event-By-Event RAS Cloud Results¶

    This section provides the information necessary to interact with the RAS output data generated by the cloud-compute of synthetic events. It describes the relevant files available after cloud compute, procedures for accessing the results and a general checklist for verification and investigation into the accuracy of the results on an event-by-event basis.

    These procedures apply to various stages of the FFRD workflow, and a formal review using a checklist is required during certain phases as described in the SOP.

    List of Files available for Investigations (with descriptions)¶

    The files in the directory \production>hydraulics>model-unit> directory (see Figure 1) include the model input files that were directly used for the cloud-compute. These are considered “the model”. Note that the files shown below are from the ‘production’ folder, but a very similar structure is also shown in the ‘validation’ area. Note that the cloud-compute uses the linux version of the RAS executable and several files that desktop RAS modelers commonly interact with are not direct inputs to cloud-compute but are still posted to this directory for reference and all to support a workflow to re-create a desktop version of the event when needed.

    The event root directory (see Figure 1.4) is shown below and includes results from each of the H&H models used for the project.

    Figure 1.4. Example Simulation Event Root Directory

    The hydraulics model output files are stored in the \production>simulations>event#>hydraulics>model-unit directory as shown in Figure 1.5 below. They include the HDF file, b-file and runtime output log. The HDF file is the primary output file the contains all RAS output and can be viewed with either RASMapper or HDFView. The b-file is small text file that contains trigger elevations for each breach in the RAS model, and can be considered an input file for the RAS event.

    Figure 1.5. Location of Hydraulics Model Output Files

    The hydrologic model ouptuts are stored in the \production>simulations>event#>hydrology>basin-name directory as shown in Figure 1.6 below. The key files

    The Exported Precip file is the “excess precipitation” runoff grid from HMS after losses are calculated and is a key output from the HMS model and is a direct input into the RAS model. As a standard in cloud-compute, the Exported Precip file is only saved in HDF format for ‘production’ simulations and the same data in DSS format is not saved in order to reduce files size storage. However, for ‘validation’ runs, the DSS format precip should be available. Note that accumulated precipitation grids are also viewable in the HDF output file from RAS.

    Although not used directly in RAS, the ‘Storms.dss’ file (location shown in Figure 1.7) is direct precipitation that was used as input into HMS and is available for reference.

    Verification of Cloud Simulated and Desktop Recreated Events:¶

    The following list contains many of the items needed to validate cloud-simulated events on a desktop computer. These items also apply to cloud events which have been replicated on desktop, as discussed in section 1.1.

    Checklist Items:

    • Check that it completed successfully (in RAS Mapper, or in rasoutput.log (Progress= 1.0)

    • Check volume accounting error as a percentage in rasoutput.log (should be less than around 3%-5%)

    • Check maximum WSE error in rasoutput.log (may be large due to large cells used in FFRD and steep terrain)

    • Check that rainfall was applied (create new result map layer > additional 2d variable > cumulative precipitation depth)

    • Check for breaches. Check that breaches occurred as they should have.

    • Check hydrographs (enable geometry form under results heading > enable sa/2d conns > right click structure > plot structure variables >hydrographs). Look for instability or other obvious problems.

    • Compare select rating curves against calibration/published rating curves

    • B file matches sampled breach elevations from failure-elevations.json file?

    • Check that BC line flows respected for ressim & HMS outputs, as shown in RAS HDF output file?

    • Check that excess precip applied properly

    • Check starting pool in RAS matches with starting pool from ResSim?

    • Check starting IC points respected & held constant during warmup?

    • Check reservoir initial condition. Does it contain a flat pool at IC? Any issues around edges of pol?

    • Check boundary condition flows are respected for ResSim & HMS outputs

    • Check that excess precipitation displays on top of mesh in RasMapper and typical pattern of depths from fluvial flooding are present 

    • Compare runtimes between cloud and desktop simulations
