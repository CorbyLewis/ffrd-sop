    Job Aid 8—Dam Representation¶

    This job aid describes dam physical and operational model representation specifically for FFRD models. The guidelines for dam and reservoir modeling methods are included in SOP Volume II and the guidelines for scoping of specific methods to apply for each dam in a basin is included in a separate job aid. This job aid document aims to provide the technical detail and processes required for implementation.

    Coordinated Dam Modeling Approach Across System of Models¶

    Consistency of the model approach across the HEC-HMS, HEC-ResSim, and HEC-RAS models for each dam is important for FFRD projects. The SOP utilizes the Dam Model Scoping Table, Model Linking Register, and Name Table to ensure each model can be developed based on coordinated approach.

    A common dataset that contains the available physical dam information, reservoir capacity information, and curated historic observed data is provided prior to model development and calibration activities and will be leveraged throughout model development.

    Initial Pool Elevations¶

    For each storm event, initial pool elevations for reservoirs should be set consistently across each model (HEC-HMS, HEC-ResSim, and HEC-RAS). The initial pools should be set according to standard techniques available in users manuals for each software package.

    For HEC-RAS models, initial conditions (IC) points are only required for dams scoped for Type A, B, or C modeling. For all HEC-RAS IC points, the fixed during warmup option should be selected so that the pool level at the start of the simulation is consistent with the selected value.

    Historic event modeling will leverage any available observed pool data as initial pool elevations. When no observed data is available, estimating the average pool level is acceptable and LiDAR elevations that capture the pool water surface may be used as an approximation.

    Stochastic events will vary initial pool levels for each event for certain dams according to the Dam Model Scoping Table. The method for selection pool elevations for stochastic simulations for each event is documented in other sections of the SOP.

    Reservoir Storage Capacity¶

    The reservoir storage capacity for each reservoir is an important aspect of the model representation for each modeled dam and must be consistent across the HEC-HMS, HEC-ResSim, and HEC-RAS models. All aspects of this representation are described in the sections below.

    HEC-HMS Storage Capacity¶

    A tabular Elevation-Storage Relationship represents the HEC-HMS storage capacity. It should be input such that is based on the best available data and is consistent with the RAS Mapper Terrain.

    HEC-ResSim Storage Capacity¶

    The HEC-ResSim storage capacity is represented by a tabular Elevation-Storage Relationship. It should be input such that is based on the best available data and is consistent with the RAS Mapper Terrain.

    HEC-RAS Storage Capacity¶

    The HEC-RAS storage capacity is represented by the RAS Mapper Terrain, which must account for underwater bathymetry. The RAS Mapper Terrain is generated by incorporating the best available data into the base terrain through GIS processes. Steps to compare RAS Mapper Terrain-based storage with Tabular estimates applied to HEC-HMS and HEC-RAS are needed to ensure consistency.

    Storage Capacity Consistency Check¶

    After the RAS Mapper based terrain has incorporated bathymetry to represent the storage, it is critical that the terrain’s elevation-volume relationship is validated against any known elevation-volume data. This could be a published elevation-volume curve, tabular data from historic documents, or National Inventory of Dams (NID) storage values. Elevation-volume curves can be computed for a given area within RAS Mapper or in the geometric schematic editor.

    To compute a RAS Mapper terrain-based storage, a separate new geometry should be created for this purpose. A new 1D storage area should be created within this new geometry with a name corresponding to the reservoir of interest. The boundary of this storage area should be close to the perimeter of the water body as indicated by the LiDAR terrain digital elevation model (DEM) (see the note below regarding automatic generation of reservoir outline polygons). This geometry with a storage area over the reservoir needs to be associated with the terrain that has bathymetric terrain modifications applied. Right-clicking the storage area allows for computing and plotting elevation-volume data for the reservoir.

    The elevation-volume curve computed by HEC-RAS from the terrain should be compared with available elevation-volume data. The terrain should be modified until there is an appropriate fit between the terrain surface’s volume and available reference data.

    The reservoir outline polygon needed to produce the storage capacity table can be created through an automated process. GIS geoprocessing tools, such as Contour, can be used to automatically generate an outline of the reservoir surface (assuming the reservoir is a flat surface in the LiDAR based DEM). If using the contour geoprocess, the base contour can be specified as slightly higher than the LiDAR indicated water surface elevation (e.g., base contour of 0.1 feet higher than LiDAR water surface). A large contour interval can also be used by the contour geoprocess so that only a single contour is created at the desired elevation near the LiDAR water surface. A contour type of contour is appropriate for this purpose (the output polyline feature can then be converted to a polygon using the Feature to Polygon geoprocess). The Processing Extent options of the Geoprocessing Environments can be used to limit the extent where contours are calculated. Some manual cleanup may be required to remove contours which do not correspond to the reservoir of interest. The Simplify Polygon geoprocess with a tolerance of approximately 1 to 2 feet can be used to reduce the number of vertices in a reservoir outline polygon.

    Storage Capacity Source Data¶

    There are several sources of reservoir bathymetry data, many of which are discussed below. Note that authoritative input data for the project will be collected during the Scoping and Data Preparation Phase and posted to the ‘basin-data’ directory for the project. These datasets will not be available for all reservoirs, and therefore they have been roughly listed in order of their importance. Judgement will be needed to ensure the best available data to estimate reservoir storage is being applied for the project. For example, if recent bathymetric surveyed data is available, it should be utilized. If survey is not available, historic documentation should be used to estimate bathymetry, and so on. It may be necessary to use information from multiple data sources to best approximate bathymetric data.

    Survey Data¶

    Surveyed bathymetric data can be the most accurate representation of reservoir bathymetry. If this data is available and reliable, it should be used as the primary source of reservoir storage model inputs. The format of this survey data will vary and may come in the form of a surface dataset (e.g., triangular irregular networks [TIN], raster, etc.), individual survey points, surveyed transects or cross-sections, or other formats. The sections below describe the typically available data, followed by processes that may be used for application of the selected source data on an individual dam basis.

    Historic Documentation¶

    Historic documents that describe the storage volume of reservoirs may be available. These documents may be available from the reservoir’s construction, or more recent studies. Common examples of these documents include Water Control Manuals, Reservoir Regulation Plans, Reservoir Sedimentation Studies, As-Built documents, and other design documentation. Ideally, these documents may provide elevation-volume or elevation-area data (either tabularly or graphically), which describes how the reservoir’s storage varies by elevation. If elevation-varied storage information is not available, storage volumes may be specified at given elevations of interest, such as conservation pool, spillway crest, top of dam, etc.

    National Inventory of Dams¶

    The NID provides a national database of dams and reservoirs and includes some basic information on their storage volumes. Normal and maximum storage are provided as attributes for all reservoirs in the NID. According to the NID Data Dictionary, maximum storage is defined as “the total storage space in the reservoir below the maximum attainable water surface elevation, including any surcharge storage.” Normal storage is defined as, “the total storage space in a reservoir below the normal retention level, including dead and inactive storage and excluding any flood control or surcharge storage.” For approximation purposes, it can generally be assumed that the reservoir storage below the LiDAR indicated water surface elevation in the DEM is equivalent to the NID normal storage, unless available data indicates otherwise.

    Historic Topographic Mapping¶

    Historic topographic mapping may be available from the period prior to the reservoir’s impoundment, providing topographic contours for the terrain beneath the reservoir. For very old reservoirs, this information may not be available, however historic topographic maps are available widely across the United States from the USGS beginning in the 1940s. The accuracy and detail of these maps varies and may be limited, but they do provide some estimate of the bathymetric surface. Data from these maps can be converted to DEMs and corrected to match a published elevation-volume curve where data is available.

    Terrain Indications¶

    When all other data discussed above is unavailable, LiDAR DEM terrain data may be useful for approximating bathymetry. For example, the storage volume of dry dams (dams which do not impound water permanently) may be accurately represented by the terrain DEM without additional information. For dams which do impound water but have no other information available regarding their storage, the elevation of the dam’s toe on the downstream side is likely a reasonable approximation for the deepest portion of the reservoir. The reservoir’s elevation is also well defined around the lake water’s edge, where depths are zero. A rough approximation regarding the reservoir’s volume can be estimated from this information alone. LiDAR DEMs may show locations where tributaries enter the reservoir body, these tributaries likely continue beneath the reservoir’s surface and any estimates of the lake’s bathymetry should attempt to capture these features. The bathymetry beneath the lake’s surface likely resembles the surrounding area, which should be well described by available LiDAR DEMs. These surrounding areas can provide context information on things like valley wall slope, pre-impoundment river channel width and meander characteristics, valley shape, and many other geomorphic features which can inform bathymetric approximations.

    RAS Mapper Terrain-Based Storage Capacity¶

    There are multiple methods that may be used to incorporate reservoir bathymetry data into the RAS Mapper terrain to accurately represent the best available reservoir capacity information. In the sections below, typical workflows and processes for incorporating reservoir bathymetry are presented. The selection of specific method for each reservoir must be made based on the available source data and by applying judgment according to engineering best practices.

    Regardless of the method applied, care must be taken to ensure any required vertical datum conversions are properly applied to the input data such that the reservoir storage representation meets the FFRD vertical datum standard.

    Survey Data Approach¶

    When an array of survey points is used as the source data, a TIN is a simple method of interpolating elevations between known points, which preserves the exact elevation at the known points. This approach is appropriate for converting point-based bathymetric survey data into a raster format which can be merged with a DEM at elevations above the reservoir pool. Compared to other interpolation methods, TIN creation has few required parameters and does not rely heavily on user judgment. However, the spatial distribution of input data points can cause errors in estimation, which will be discussed in the next section. This guide references tools in ArcGIS Pro, however, similar methods may be available in other GIS software.

    GIS Processing Steps¶

    First, if necessary, a datum conversion should be applied such that the vertical datum of the bathymetric survey data matches the vertical datum of the DEM that it will be merged with. Next, the bathymetric survey data should be imported into ArcGIS Pro as a point feature class, using the XY Table to Point tool if the data was delivered in tabular format. Next, using a DEM as input, a polygon of the approximate reservoir extents at the time of the bathymetric survey should be generated. This can be done using the Contour tool. To create a single contour polygon, the base contour should be set to the desired pool elevation, and the contour interval should be set to a very large number beyond the range of the DEM values.

    Figure 1. Creating a Single Contour Polygon

    This reservoir extent polygon will be used as a constraint on the TIN creation. To ensure that no observed points are clipped out, the contour elevation should be set to several feet above the elevation at the time of data collection.

    Optionally, to ensure that the slope of the TIN is logical, a thalweg line can be applied during TIN creation as a breakline. A breakline forces the process to consider areas on opposite sides of the breakline as independent from each other. Thalweg data may be provided with survey data or may be created using other methods. Breaklines are useful for constraining and refining interpolation between survey points and may be required to adequately reflect the reservoir’s bathymetry.

    With point data, a reservoir extents polygon, and (optionally) a polyline of the thalweg, the Create TIN geoprocessing tool can be used. The point data should be input as Mass Points with the height field set to match the elevation field in the Attribute Table. The reservoir extents polygon should be used as a Hard Clip. The thalweg should be used as a Hard Line. The output coordinate system should match the DEM, and the Constrained Delaunay option is up to the user.

    Next, the TIN to Raster geoprocessing tool can be used to convert the TIN into raster format. The sampling distance should be Cell Size, and the Sampling Value should match the cell size of the DEM (this should be precise, and may need to be copied and pasted from the DEM’s Properties dialog box). Using the Natural Neighbor method versus the Linear method results in very similar storage capacity relationships, but the Natural Neighbor method is more computationally robust, producing a smoother raster surface. Finally, in the Environments tab, the Snap Raster should be set to the DEM to enable seamless merging of the bathymetry raster and the DEM.

    Next, as an optional step, the resulting bathymetric surface can be merged with the LiDAR based DEM to produce an elevation-storage relationship for all reservoir pools. This step should be done carefully to minimize any breaks in the data or large jumps in elevation, which can occur if there is a discrepancy between the bathymetric data and the DEM due to the uncertainty and slight estimation error that exists in all terrain data. The recommended approach is to create a contour at the exact pool elevation which existed during the bathymetric survey and use Extract by Mask to mask the TIN to Raster result to these extents. The Snap Raster should be the DEM when using Extract by Mask. Then, Mosaic to New Raster can be used to merge the result with the DEM, setting the Mosaic Operator such that the values from the bathymetry raster will be carried forward wherever they exist, and the values from the DEM will be carried forward outside the extents of the bathymetry raster. In the Environments tab, the Processing Extent of the Mosaic to New Raster tool can be set to a display extent that covers the entire reservoir, which is necessary to reduce file size if the source DEM covers a much larger area than needed. If the result is unsatisfactory, a different reservoir pool extent could be used in Extract by Mask to mask the TIN to Raster result, and Mosaic to New Raster could be applied again using the modified TIN to Raster result.

    Special Considerations and Potential Error Sources¶

    For the greatest accuracy in TIN creation, the ideal situation is for known data points to be plentiful and randomly distributed through space. This condition would require the use of multibeam survey equipment. However, bathymetric data is sometimes collected along defined paths in a reservoir using a single beam echo sounder, which leaves large gaps between collected data. Figure 2 shows an example of this condition, which requires special considerations to be made during interpolation.

    Figure 2. Large Gaps between Collected Data due to Single Beam Echo Sounder Data Collection

    In this example, data was collected in cross sections and transects, with very small point spacing along the cross section and very large point spacing between cross sections. This distribution of points creates error in the final result. This error occurs because, in a reservoir, elevations are typically changing very gradually parallel to the flow direction of the original stream channel, but elevations change rapidly perpendicular to flow. This feature of the data is called anisotropy. In TIN interpolation, there is no way to correct for this condition, so interpolated points are generated based on nearby values in all directions. After the TIN was converted to a raster, the result was that elevations were overestimated in areas far from observed points. Figure 3 shows the locations of cross sections where the elevations are correct, with overestimated elevations (in darker shades of green) in between the cross sections.

    Figure 3. Cross Sections with Correct Elevations with Overestimated Elevations between Cross Sections

    The overestimation in this example was more severe due to the narrow, steeply-sloped valley in which this reservoir exists. A wider reservoir with flatter slopes would be less prone to this particular error. Other approaches, such as the Topo to Raster geoprocessing tool can provide a more suitable interpolation between data points in some cases, but each approach has limitations. In general, the result of the TIN creation process should be carefully examined for reasonableness. If there is clear overestimation or underestimation of elevations, another approach should be considered. Additional breaklines can be used to guide interpolation and reduce error. In some cases, if no one suitable approach can be found, engineering judgment may be necessary to combine the results of multiple approaches.

    Approximate Approach¶

    When elevation-volume data is available, but a bathymetric terrain surface is not, HEC-RAS terrain modifications can be used to modify the LiDAR DEM terrain surface and approximate the given elevation-volume curve. This approach involves using multipoint polygon terrain modifications and control points to approximate a bathymetric surface in RAS Mapper. This approximate bathymetry is iteratively modified until an acceptable match is achieved between the published elevation-volume data and the elevation-volume computed from the terrain surface. This process is similar to the one described in Detailed Approach section below. Figure 7 displays an example comparison of what can be achieved by iteratively approximating an elevation-volume curve by using HEC-RAS terrain modifications. Additional refinement of terrain modifications could be performed to better improve the fit between the two curves. Refer to the ‘Storage Capacity Consistency Check’ section for additional details on how to compute an elevation-volume curve from a terrain surface within HEC-RAS.

    Historic contour data can be used to inform the shape of the bathymetric terrain, but is not always necessary. This approximate approach differs from the approach discussed in Detailed Approach section in which the terrain surface is corrected to closely match published elevation-volume data. While this approximate approach may not perfectly reproduce the curve, it is an efficient process and is appropriate for smaller impoundments that do not pose a high breach risk to downstream communities. Engineering judgement and team consensus will be required to determine which bathymetric estimation approach is appropriate for a given reservoir and when a given match between modeled and published volume estimates is acceptable.

    Figure 7. Example comparison of Published and Approximated Elevation-Volume Curves

    Terrain modifications in RAS Mapper take advantage of vector editing tools to create points, polylines, and polygons to modify an existing HEC-RAS terrain. These tools are useful for incorporating bathymetry into a terrain surface, as well as many other uses. Additional information regarding the use of terrain modifications is disused in this document, but also in the RAS Mapper User Manual.

    Detailed Approach¶

    For large reservoirs which pose significant breach risk to downstream communities, it may be merited to pursue a detailed approach to estimate reservoir bathymetry that reproduces a published elevation-volume curve accurately. This detailed approach may also be required for reservoirs where HEC-ResSim operations are being simulated. This is required because HEC-ResSim specifies elevation-volume relationships using published tabular data, and any disagreement between reservoir storage in HEC-RAS and HEC-ResSim will result in disagreement of computed water surfaces in the reservoir.

    The detailed approach consists of creating an approximate bathymetric terrain surface and then correcting that surface such that it reproduces a published elevation-volume curve. These two steps are described below.

    Creating a Bathymetric Surface¶

    There are several ways to create an approximate bathymetric terrain surface. One, which has been employed successfully on the FFRD pilot projects, is to generate a terrain surface from historic topographic maps from prior to the reservoir’s impoundment, which are widely available across the U.S. . Note that for historic topography to be useful in this process, the historic maps must include contour data in the reservoir pool, which is common in maps created prior to reservoir impoundment.

    The process to convert historic topographic mapping to a terrain which can be utilized directly in HEC-RAS is described in Table 1. This process involves tracing topographic map features such as contour lines and stream centerlines. It may be possible to automate this procedure such that manual digitization of contour lines is not required, however for smaller reservoirs where contour data is not particularly dense, manual digitization often seems to be the most efficient option.

    The process is designed around the use of ArcGIS Pro and utilizes either the 3D Analyst Tools or Spatial Analyst License, specifically the Topo to Raster geoprocessing tool. It is recommended to review available documentation for the Topo to Raster tool prior to use to understand its data requirements and limitations. Some resources regarding the Topo to Raster tool include:

    • https://pro.arcgis.com/en/pro-app/latest/tool-reference/3d-analyst/how-topo-to-raster-works.htm

    • https://pro.arcgis.com/en/pro-app/latest/tool-reference/3d-analyst/topo-to-raster.htm

    • https://support.esri.com/en-us/knowledge-base/why-does-the-topo-to-raster-tool-generate-inaccurate-el-000012375

    This process should produce a terrain surface DEM that can be directly incorporated into HEC-RAS, however the terrain surface generated will likely not agree with other sources of elevation-volume data, such as that commonly available in water control manuals. The next section describes how a terrain surface can be corrected to improve agreement between a terrain surface and given elevation-volume data.

    Correcting a Bathymetric Surface to Reproduce an Elevation-Volume Curve¶

    Elevation-volume, and specifically elevation-area relationships, can be used to modify a given terrain surface such that it reproduces the input elevation-volume relationship. It should also be noted that this volume correction process can be applied to a diverse range of initial terrain surfaces, and not only surfaces created from historic topographic mapping as discussed in the previous section. Initial terrain surfaces should resemble the true bathymetric surface as well as is practical for best results. The volumetric correction process can be performed in a script or spreadsheet using linear interpolation methods to return values from a given elevation-area curve corresponding to a specified contour surface-area. If done in Excel, a custom linear interpolation function is very useful.

    The steps to perform a volumetric correction process on a terrain is described in Table 2. This process generates elevation contour polygons from a terrain surface (step 6) and calculates the surface area associated with each contour (steps 7 through 9). Once the area of each terrain contour is known, the corresponding elevation can be looked up from published elevation-area data (step 10). A linear interpolation script is useful for this task; this is the corrected elevation attribute. Now each terrain contour has attributes of surface area, elevation (as computed from the historic bathymetry estimate), and a corrected elevation looked up from published data.

    These elevation-corrected contours can be used as input data to generate a new terrain surface (steps 11 through 12). The elevation-volume relationship of the new surface should be verified (step 13). If needed, the process can be iteratively performed to improve the fit between published and modeled volume data (step 14). Finally, the process should create a terrain surface suitable for use in HEC-RAS, which reproduces elevation-volume relationships that will be used in the paired HEC-HMS and HEC-ResSim models (step 15).

    As noted previously, this volumetric correction approach relies on elevation-surface-area relationships. Refer to Section 1.3.6.2, and specifically Figure 6, for an example calculation of how to convert an elevation-volume dataset to an elevation-area dataset. Note that the surface-area calculated using this process corresponds with the average elevation data shown in column D of Figure 6, not column A. Also refer to section 1.3.4 for the process to verify a terrain surface’s elevation-volume relationship in HEC-RAS (step 13).

    Storage Approximation with Basic Storage Attributes¶

    In many cases, complete elevation-volume data may not be available for a given reservoir. In these instances, the NID may have some limited information on the reservoir’s volume. One attribute of the NID database is the Normal Storage. For approximation purposes, it can generally be assumed that the reservoir storage below the LiDAR indicated water surface elevation in the DEM is equivalent to the NID Normal Storage, unless available data indicates otherwise. This single storage value can be used to estimate and validate a complete elevation-volume curve generated using HEC-RAS terrain modifications. Other terrain indications, discussed in Section 1.3.5.5, can be useful in approximating bathymetric terrain features.

    The process involves iteratively creating HEC-RAS terrain modifications to estimate a bathymetric surface until the elevation-volume curve intersects the normal storage volume indicated in the NID at the water surface elevation indicated by the LiDAR terrain data. Figure 9 displays an elevation-volume curve computed in HEC-RAS using this approximation technique as well as the published normal storage point from the NID. Note that in this case the curve compares relatively well to the published value after only a single attempt to estimate bathymetry. The published NID value does indicate slightly more volume that estimated by the terrain (approximately 4 percent additional storage at LiDAR surface elevation of 519.3 feet). In this case, the terrain modifications could be refined to increase volume slightly to achieve better agreement between the terrain and NID volume.

    One convenient way to achieve better agreement between the terrain and NID volume is to calculate the vertical difference in elevation between the elevation-volume curve and the NID storage value. In the example shown in Figure 9, the curve is approximately 0.2 feet above the published storage value. To adjust this curve, all the terrain modification control points can have their assigned elevations lowered by 0.2 feet and the curve should then intersect the published storage volume. This vertical adjustment assumes the overall shape and curvature of the elevation-volume data is reasonable.

    Storage Approximation with No Supporting Data¶

    In cases where there is no known volume information on a reservoir, such as when the NID is missing a normal storage attribute, there is still a need to estimate bathymetry if breach analysis will be performed on that reservoir. In that case, the approach utilized in Section 1.3.6.3 should be followed, although there will be no known storage point to validate the bathymetry estimate against. Best judgement should be used to approximate bathymetry in these cases. Refer to Subsection 1.3.5.5 for more information on how features from a LiDAR based DEM can inform bathymetric estimates.

    Reservoir Outflow¶

    Operational releases from reservoirs are a key aspect of model representation. For FFRD models, the method applied for each dam in each of the models (HEC-HMS, HEC-ResSim, and HEC-RAS) will be guided by the selections made in the Dam Scoping Table. Modeling development decisions should always be made with an understanding of the interaction of other models in the FFRD system. The sections below provide additional modeling guidance of the implementation of the selected methods.

    HEC-HMS Reservoir Outflows¶

    For dams that are selected to represent a reservoir element in HEC-HMS, the following guidelines, techniques, and troubleshooting tips are offered. Note that the Dam Scoping Table will provide direction for HEC-HMS modeling using Simple Reservoir Operations, Complex Operations, or no inclusion in the HEC-HMS model.

    Un-modeled Reservoirs¶

    To account for the attenuation of flows through reservoirs not modeled in HEC-HMS, HEC-HMS may account for their presence with adjustments to the unit hydrograph parameters, increasing the time of concentration or storage coefficients. This approach can improve historic event calibration in HEC-HMS and improve the accuracy of HEC-HMS output data that passes to HEC-ResSim and HEC-RAS models.

    Simple Reservoir Operations¶

    Simple reservoir operations include one to one storage and discharge relationships, uncontrolled outlets, and ungated spillways. Typically, if the outlets and spillways have simple rating curves not dependent on inflow or seasonality, the dam can be modeled with simple procedures in HEC-HMS. A single outflow curve is the simplest hydraulic routing method available in HEC-HMS. If outlets are gated or controlled, simple operations can still be used by analyzing observed data and estimating an average outflow for a given elevation or storage condition. Guidance on creating a single outflow curve can be found in the Outflow Curve Option for an HEC-HMS Reservoir Element guide. Storage-discharge and elevation-storage relationships should be uniform across the HEC-ResSim, HEC-HMS, and HEC-RAS models (to the extent possible).

    Complex Reservoir Operations¶

    For more complex reservoir operations, the use of HEC-ResSim to model operating schedules, rules, and downstream constraints may be scoped for the FFRD project, however in some cases rule-based operations in HEC-HMS may by scoped for the project to provide an accurate and more efficient approach. When HEC-HMS rule-based operations are scoped for use as the authoritative reservoir outflow model, the following guidelines should be used.

    Storage Objective Patterns¶

    Storage patterns can be used based on published reservoir schedule guide curve or developed based on observed historical elevations converted to storage. Taking the 50 percent exceedance value of a duration analysis of the observed period of record of daily elevations can serve as a guide curve of storage objectives. The relationship should be input to the HEC-HMS model as a Paired Data–Storage Pattern table of calendar days and corresponding volume target.

    Zones¶

    Zones should be established to implement rules and store objective patterns that may change throughout the range of elevations. A Paired Data–Daily Elevation Pattern is needed to define the maximum elevation of the zone. This can be a constant elevation, or the zone elevation limits can vary throughout the calendar year.

    Release Rules¶

    Simple minimum and maximum release rules should be utilized first before adding release rate of change rules. Typically, this would include maximum release rules in a Zone to manage downstream capacity constraints or minimum release rules to better match gradual release patterns and smooth our spike releases. Release rate of change rules can be used. It is recommended to test shorter continuous simulations (approximately two water years) to see how the release rules perform and add complexity as needed.

    Downstream Constraints¶

    Downstream flow constraints are very computationally expensive and using a downstream constraint rule should be avoided in HEC-HMS. If a downstream flow constraint is prescribed in reservoir operations, this constraint should be managed to the extent possible with a maximum flow release from the reservoir that takes into account additional downstream flows that will accumulate at the downstream constraint point.

    Gated Outlets¶

    Gated outlets should be added as General Outlets. The minimum curve should be set to zero and the maximum curve will represent the maximum discharge possible or prescribed for a given elevation. The minimum time between adjustments should be sufficiently short so that the reservoir does not bottom out in between adjustments and not so short that the reservoir pool begins to oscillate unstably. It is recommended to start with a minimum adjustment time of 60 minutes and adjust as needed.

    Potential Modeling Obstacles¶

    Modeling reservoirs in HEC-HMS can be a trial-and-error process, the modeler should start with the simplest modeling method and add complexity as needed to best represent reservoir operations. During that process there are some common pitfalls.

    Reservoir Bottoming out¶

    The modeler should check that the minimum time between adjustments is not too long. If this is set too large, the gate settings may not be able to adjust to the prescribed storage pattern quickly enough. This is especially true for reservoirs with very low storage volumes near the bottom of the pool. Another potential workaround is to set a fairly low minimum release near the bottom of the pool or a constant storage pattern to ensure minimal or no releases occur during times where available storage is very low.

    Long Runtimes¶

    Avoid using downstream constraints and, to the extent possible, any rules that necessitate a decision between two outlets on where to make releases. This would include adding main and auxiliary outlets and having a minimum or maximum release rule from auxiliary and main outlets.

    Abrupt spikes in releases or lack of attenuated releases¶

    Spikes in releases can be smoothed by applying a maximum rate of change release rule or by instituting a minimum release rule to allow releases to be more attenuated instead of, for instance, peaking at 1,500 cfs outflows and immediately dropping to 0 cfs outflows once the guide curve is matched.

    HEC-ResSim Reservoir Outflows¶

    For dams that are selected for operations modeling in HEC-ResSim, the FFRD core standards are described in SOP Volume II and the guidelines, techniques, and troubleshooting tips that follow supplement those standards.

    HEC-ResSim Scoping of Approach¶

    In the reservoir operations model, various levels of detail are available depending on the reservoir characteristics. As reservoirs and their operations increase in size and complexity, additional modeling techniques are required. Reservoir modeling decisions made during the scoping and data preparation phase of the FFRD project. The following topics will be clearly scoped and communicated through a combination of the Dam Scoping Table, Name Table, Model Linking Register, and Scoping and Data Preparation Report.

    • HEC-ResSim modeling approach for reservoir routing operations (detailed reservoir operations model, hydrologic model, or not at all)

    • HEC-ResSim modeling approach for each unique outflow feature at each dam (i.e., spillways separate from other outlets?)

    • HEC-ResSim Modeling approach for overtopping surfaces (dams and ungated spillways)

    • Determination of any locations where special treatment or coordination is needed between the hydrologic, hydraulic, and/or reservoir-operations model for consistency in modeled inflow, outflow, and reservoir elevation.

    HEC-ResSim Model Domain¶

    The HEC-ResSim model domain covers all reservoirs operating based on conditions elsewhere in the watershed. Even if each reservoir is operating independently of the others, a single HEC-ResSim model is preferred, however, in special cases it may be necessary to create several smaller headwater HEC-ResSim models for small reservoirs that cannot be modeled at the same timestep as larger reservoirs downstream in the system due to model instabilities and mass balance issues that occur when small reservoirs use too large of a timestep.

    Modeling of Overtopping Surfaces¶

    Dam overtopping, including for events larger than design storms, must be accounted for in the HEC-ResSim model and in Modeling Linking Register.

    Diversions¶

    When included in the scope, HEC-ResSim diversions should be added at major canal and diversion systems. These can either be connected to a downstream location or used to remove water from the system entirely. In heavily irrigated watersheds with many small diversions, they are lumped to the nearest downstream junction.

    Reservoir Routing Consistency¶

    For dams modeled with the Type A treatment consistency of the reservoir routing results between HEC-RAS and the model sourced for outflows (HEC-ResSim or HEC-HMS) could be carefully evaluated. The FFRD model approach includes an inherent difference between the modeling of inflows to the reservoir between the models as HEC-RAS routes HEC-HMS-generated excess precipitation and baseflows over a 2D grid while HEC-HMS applies the unit hydrograph techniques to transform runoff. While the calibration process ensures close alignment between the model results, differences that persist may result in reservoir routing problems in HEC-RAS.

    When the reservoir inflow from HEC-ResSim or HEC-HMS is out of sync with HEC-RAS simulated inflow, the HEC-RAS reservoir pool elevation hydrographs can diverge from the other models. If these differences are large, it is possible that the specified outflows for use in HEC-RAS could cause the reservoir to go dry and result in stability issues. A related issue that can arise is that a breach may be triggered based on the sampled trigger elevation for a stochastic event in the HEC-ResSim model, while the HEC-RAS model may not reach the trigger elevation resulting in an inaccurate routing of the breach outflows.

    These pitfalls are avoided by ensuring that HEC-HMS and HEC-RAS calibration efforts are coordinated such that flood routing for the historic events are well aligned. Other recommended checks during the conformance phase of the project include a comparison of simulated minimum pool and peak pool elevations between the models and spot check comparisons of event-level pool elevation hydrographs.

    Dam Breach Representation¶

    The workflows and other details regarding the dam breach approach are included in SOP Volume II and a separate job aid. However, the handling of reservoir breaching in HEC-HEC-ResSim for stochastic events requires special consideration, and an understanding of the interactions with the other models in the system is needed and described here. For dams modeled as Type A in the hydraulic model, the release of water through the breach is calculated by the hydraulic model based on the dam breach parameters (size and timing of breach opening), however a separate specified outflow boundary condition is also directly linked to the simulated HEC-ResSim releases. If not addressed, the HEC-RAS model could release water based on breach outflow and large HEC-ResSim operational releases that to not account for the falling pool caused by the breach.

    To address the issue, HEC-ResSim will breach by setting outflows to equal inflows when a pool above that trigger elevation is reached. The breach trigger elevation is sampled for each event and shared between HEC-ResSim and HEC-RAS. In the HEC-RAS model during that same event, the breach will be triggered at the same instant based on the HEC-RAS routed pool elevation and release a larger volume of water through the breach. Actual outflow that the HEC-RAS model will see is HEC-ResSim Releases and HEC-RAS computed breaches.

    There are a few imperfections in this method:

    • HEC-RAS and HEC-ResSim Simulated Pool Hydrographs may not track perfectly due to the timing of inflows. HEC-ResSim received inflows routed via HEC-HMS, while HEC-RAS simulates 2D flow over the mesh using via HEC-HMS excess precipitation and baseflow outputs

    • If pool hydrographs do not track well together, breaches will not trigger at the same time.

    • Breaches at upstream dams may trigger breaches at downstream dams. HEC-ResSim will not be able to account for this fact since the large volume in a reservoir is not actually released after a breach.

    • After a breach occurs, HEC-RAS may be releasing too much flow especially if inflows are high at time of breach.

    • It is possible HEC-RAS numerical stability issues will occur if HEC-ResSim inflows are forced out of a reservoir that is dry after a breach has released all water.
