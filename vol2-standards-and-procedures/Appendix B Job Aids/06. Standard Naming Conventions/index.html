    Job Aid 6—Standard Naming Conventions¶

    For successful integration of the models, all models and directories must use consistent naming conventions. This Job Aid defines the standard naming conventions for watershed level model data as well as for each constituent model. The more detailed naming conventions of folders and key files in the standard directory structure is provided in the FFRD demo directory, which is referred to in Job Aid 07, Model Library Use.

    A comprehensive naming convention table will be created during the Scoping and Data Preparation Phase and provided to modelers prior to the start of the Calibration Phase and will follow the requirements and guidelines in the sections below. This table will reinforce full-basin consistency, avoid duplication, and serve as the key reference for model linking that will occur during the Conformance Phase.

    Global Naming Convention Requirements¶

    • Naming conventions will use only valid American Standard Code for Information Interchange (ASCII) characters.

    • Uppercase characters will never be used (i.e. use all lower-case characters).

    • Spaces will never be used.

    • Hyphens (-) will be used for separating words to improve human readability.

    • Field Delimiters: Underscores (_) will be used as a universal field delimiter to separate individual fields contained within folder names, file names, and model element names.

    • Pre-defined formulas for the sequence of fields in the names for the vast majority of model elements are defined in the sections below and shall be followed.

    General Structure of Names¶

    Rules for naming conventions are outlined in the sections that follow. For each object included, a formula is provided to instruct users on the proper naming in a format that clearly and consistently defines the fields included in each name.

    The rules are presented using the following terminology:

    • fields (separable parts of the name)

    • delimiters (the character used to separate individual fields)

    • formulas (derivation of the object names build from fields and delimiters)

    • examples.

    Literal text that does not vary is shown in the formulas in between single quotations (‘literal-text’). In some cases, an optional field is included in formulas to account for edge cases and are included in square brackets.

    Basin-Name Examples¶

    Since the FFRD work is based on the Hydrologic Unit Code (HUC)-4 level of the National Hydrography Dataset, or subregions, the basin-names referred to within this report generally relate to the subregion names. All delimiters in basin names shall be dashes. No periods, apostrophes, commas, or other symbols shall be used. Many HUC-4 basins are named with a dash to separate different major rivers within the basin. For this convention those dashes will remain, but all spaces will also be substituted with dashes. Table 2 lists example HUC-4 basins and how they shall be named to follow this convention.

    Table 2. Example Hydrologic Unit Code 4 Basin Naming

    HUC-4 Subregion Code HUC-4 Subregion Name FFRD basin-name
    0702 St. Croix-Trout-Kettle st-croix-trout-kettle 
    1302 Middle Rio Grande middle-rio-grande
    1703 Columbia-Lower Snake columbia-lower-snake
    0307 Altamaha-St. Marys altamaha-st-marys

    Project Repository Name¶

    The FFRD watershed study is organized within the Model Library within a repository named using the FFRD descriptor followed by the basin name text. The basin name will typically match the HUC-4 name in the Watershed Boundary Dataset (WBD).

    • formula: ‘ffrd’ _ basin-name

    • examples: ffrd_big-horn, ffrd_trinity

    Digital Elevation Model Names¶

    The base DEMs created during the Scoping and Data Preparation Phase will be named according to the formulas below. Which formula to use is dependent on whether the delivery is provided as a single DEM for the full study basin or broken out into smaller pieces according to the hydraulic model units.

    The naming convention for full study basin DEMs provided for the entire basin will follow the formula below.

    • Formula: ‘dem’ _ basin-name _ resolution [ _ ‘hydro-enforced’]

    • Example: dem_big-horn_30m, dem_big-horn_1m

    The naming convention for the DEM produced for individual hydraulic model units follows the formula below.

    • Formula: ‘dem’ _ model-unit _ resolution [ _ ‘hydro-enforced’]

    • Example: dem_west-fork-creek_1m, dem_east-fork-creek_1m

    RAS Mapper Terrain Names¶

    The terrains created by RAS Mapper produced for each model unit will follow the formula blow.

    • Formula: terrain _ model-unit _ resolution

    • Example: terrain_cedar-creek _4ft

    Storm Names¶

    Each storm in the storm catalog is stored in an individual HEC-DSS file, which is named based on the storm start date, the duration, data type, storm type, and the total cumulative precipitation ranking of the storm.

    Note

    r is used in place of t, which is commonly used as a parser for time in ISO formats.

    • formula: event-start-date _ storm-duration _ storm-type _ storm-rank

    • example: 19760102_72hr_ st4_r001

    Storm DSS Path Names¶

    HEC-DSS files use a path name internally to organize data in a tidy way. There are some required path name parts and some free-form parts to a valid .DSS path. A path has six parts labeled A through F. The C, D, and E parts have strict requirements for grid data. A and B are free form but have a small character limit. F is usually the catch-all path part due to its longer character limit.

    The convention for the storms catalog data is as follows:

    /\$CRS\$resolution/\$basin/PRECIPITATION/\$start-date/\$end-date/\$data-source/

    • A part: The Coordinate Reference System (CRS) should be SHG for the Standard Hydrologic Grid. The resolution should match the source precipitation data resolution (e.g., 4K for 4-kilometer resolution).

    • B part: Use the same short basin name that is used in the file name.

    • C part: For gridded precipitation, always use PRECIPITATION.

    • D part: For gridded data, this part indicates the data’s recording start time in ddBBBYYYY:HHMM format, where BBB is the first three letters of the month name, and dd is a zero-padded date.

    • Each 1-hour precipitation grid has its own path name. A storm .DSS file will contain 72 path names. The D part of the first path will match the start date in the filename.
    • E part: For gridded data, this part indicates the data’s recording end time in the same format as the D part. For hourly data, this should be one hour later than the D part.
    • The E part of the last path will match the end date in the filename.
    • F part: indicate the source of the precipitation data (e.g., AORC).

    An example of a valid .dss path is below.

    /SHG4K/TRINITY/PRECIPITATION/06OCT1982:0000/06OCT1982:0100/AORC/

    Hydrologic Model Names¶

    Project Name¶

    The HEC-HMS project name is based on the basin name. In cases where multiple hydrologic models are created for a single study, an optional field may be added using an integer starting with 1, for each model created. Project names and titles must be identical.

    • formula: basin-name [ _ integer ]

    • examples:

    • trinity

    • rock-river_1

    • rock-river_2

    Project Components¶

    An HEC-HMS project is defined using several files which contain information pertaining to the land surface processes/parameters, meteorologic conditions, computational settings, and simulations.

    Basin Models¶

    Basin Models must be named using the historic event time period and/or purpose.

    • formula: event-name

    • examples:

    • apr-1997

    • por

    • sst

    Meteorologic Models¶

    Meteorologic Models must be named using the historic event time period and/or purpose.

    • formula: event-name

    • examples:

    • apr-1997

    • por

    • sst

    Control Specifications¶

    Control Specifications must be named using the historic event time period and/or purpose.

    • formula: event-name

    • examples:

    • apr-1997

    • por

    • sst

    Computes¶

    Computes (i.e., simulation runs, forecast alternatives, etc.) must be named using the historic event time period and/or purpose.

    • formula: event-name

    • examples:

    • apr-1997

    • por

    • sst

    Shared Component Data¶

    Hydrologic simulation requires a wealth of data that is used as initial conditions, boundary conditions, or parameters. Time series, paired data, gridded, and terrain are all examples of shared component data.

    Time-Series Data¶

    Time series data in the hydrologic model must be named to match the location and time step of the data.

    • formula: location _ time-step

    • examples:

    • trinity-river-at-dallas_1-day

    • bridgeport-lake-inflow_1-hour

    • shasta-lake-elevation_1-hour

    • sayers-dam-outflow_15-min

    Paired Data¶

    Paired data in the hydrologic model must be named to match the location and type of data.

    • formula: location _ data-type

    • examples:

    • big-bear-lake_storage-outflow

    • big-bear-lake_elevation-storage

    • delaware-rv_r020_storage-outflow

    • wb-susquehanna-rv_r150_xs

    Grid Data¶

    Gridded data in the hydrologic model that is being used during the calibration phase must be named to match the data source and historic event time period.

    • formula: data-source _ data-type

    • example: aorc_apr-1990

    Hydrologic Terrain¶

    Terrain data in the hydrologic model that is being used during the calibration phase will be sourced directly from the 30-meter resolution DEM created during the Scoping and Data Preparation Phase and the name will not be altered when loaded in the HEC-HMS project.

    • formula: data-source _ horizontal-resolution

    • example: ned-30m

    Basin Elements¶

    Basin elements consist of subbasins, reaches, reservoirs, junctions, diversions, sources, and sinks. These elements are used to describe the physical properties of the watershed and the topology of the stream network. Basin elements are organized within basin models.

    Subbasins¶

    Subbasins must be named using a reference name following the HEC-HMS convention, suffixed with an s, followed by the HEC-HMS unique element ID. The unique ID for subbasins will follow a numbering system that increases from downstream to upstream, starting at 010 and incrementing by 10 (e.g., 010, 020, 030, etc.).

    When a subbasin element is used as a comparison to observed data at a gage, the description field in the model should be used to document the authoritative name of the gage.

    • formula: reference-name _ ‘s’ unique-id

    • example: white-rock-ck_s040

    Reaches¶

    Reaches must be named using a reference name following the HMS Convention, suffixed with an r, followed by the HMS unique element ID. The unique ID for reaches will follow a numbering system that increases from downstream to upstream, starting at 010 and incrementing by 10 (e.g., 010, 020, 030, etc.). When multiple reaches are used between two junctions, numbers should be incremented by 1 (e.g., 010, 011, etc.).

    When a reach element is used as a comparison to observed data at a gage, the description field in the model should be used to document the authoritative name of the gage.

    • formula: reference-name _ ‘r’ unique-id

    • example: trinity-rv_r070

    Junctions¶

    Junctions are complex and require special consideration. In general, junctions should be named using a reference name following the HEC-HMS convention, suffixed with a j and the HEC-HMS unique element ID. In cases when the junction is directly associated with the outlet of a subbasin, the same unique ID should be used for both the subbasin and junction. This will work in many cases where unnamed tributaries join main stems of larger water bodies. In cases where larger water bodies connect, the plus sign can be used to identify the confluence using a second optional reference name. For cases where streams and rivers flow into reservoirs, in should be used as the second reference name. Likewise, junctions that represent outflow from a reservoir should use out as the second reference name.

    When a junction element is used as a comparison to observed data at a gage, the description field in the model should be used to document the authoritative name of the gage.

    • Primary delimiter: underscore (universal for all FFRD models and filenames)

    • Secondary delimiter: Optional plus-sign for additional reference names

    • formula: reference-name [+ reference-name] _ ‘j’ unique-id

    • examples:

    • west-fork_j090

    • elm-fork+clear-ck_j050

    • nid_tx05966_in_j070

    • nid_tx05966_out_j010

    Reservoirs¶

    Reservoirs will be named using ‘nid’ followed by National Inventory of Dams Identifier (NID ID). the name may also include the human-readable dam name and “Other Structure ID” from the NID as the following fields. These fields are sourced from the NID. Note that guidance for modeling methods for individual dams will be created as part of the Scoping and Data Preparation Phase.

    The reservoir description field will include the authoritative NID “Name” field.

    • formula: ‘nid’ _ nid-id [ _ dam-name ] [ _ other-structure-id ]

    • examples:

    • nid_tx05966

    • nid_mn00584_springfield-dam_s002

    • nid_mn00584_springfield-dam

    Diversions¶

    Diversions must be named using a reference name following the HMS convention, suffixed with a d, followed by the HEC-HMS unique element ID

    • formula: reference-name _ ‘d’ unique-id

    • example: lost-creek-diversion_d001

    Sinks¶

    Sinks must be named using a reference name following the HEC-HMS convention.

    • formula: reference-name

    • example: trinity-bay

    Sources¶

    Sources must be named using a reference name following the HEC-HMS convention and prefixed with source.

    • formula: ‘source’ _ reference-name

    • example: source_trinity-bay

    Reservoir Operations Model Names¶

    Watershed Name¶

    The HEC-ResSim watershed name is also the name of the directory containing all elements of the model—it is essentially the model name. When creating a new HEC-ResSim model for this FFRD work, it shall be named according to the following:

    • formula: “ffrd_”+basin-name

    • examples:

    • Trinity River Basin = “ffrd_trinity”

    • Rock River Basin = “ffrd_rock-river”

    • Altamaha-St. Marys = “ffrd_altamaha-st-marys”

    • Rio Grande-Elephant Butte = “rio-grande-elephant-butte”

    If an existing watershed is used, the watershed name can be changed within the file system by changing three items in coordination while HEC-ResSim is not running: the main directory name, the project file (.wksp) name, and the first line of the .wksp file.

    Alternative Names¶

    HEC-ResSim alternative names are limited to only ten characters, so the naming must be succinct.

    For the Calibration Phase modeling, several alternatives will be created, including the guide curve operations, the pulse flow test, and the calibration events. Each alternative shall include an informative description. The calibration events shall be named using the following formula:

    • Formula: [first letter of the month during which the event begins]-[4-digit year]

    • Examples:

    • April 1990 = “a-1990”

    • October-November 2015 = “o-2015” or “o-n-2015”

    For the Conformance/Production Phase models, the HEC-ResSim alternative name will use a standard name:

    • ‘fema_ffrd’

    Run Names¶

    The modeler does not create run names, but scripting with HEC-ResSim and HEC-ResSim output uses run names that are based on the ten-character alternative name appended with a zero for normal runs. HEC-ResSim has the option to create trial versions of runs, which are numbered sequentially, however trials shall not be used in the FFRD modeling. If the alternative name is fewer than ten characters, HEC-ResSim appends dashes until it gets to ten characters before appending the final zero character.

    • Examples: “a-1990----0”, “o-2015----0”, “o-n-2015--0"

    Simulation Names¶

    Every simulation shall have a short description to indicate its purpose. For example, “Calibration Event for Apr-May 1990”, “conservation of mass test run”, or “FEMA Future of Flood Risk Data cloud compute simulation”.

    For the simulations of calibration events in the Calibration Phase model, the HEC-ResSim simulation name will follow the following formula:

    • Formula: “c_” alternative name (the “c” stands for calibration)

    • Examples: “c_a-1990”, “c_o-2015”, “c_o-n-2015”

    For the Conformance/Production Phase models, the HEC-ResSim simulation name will use a standard name:

    • ‘fema_ffrd’

    Reservoir Names¶

    Where possible, reservoir names will directly correspond with the reservoir element names applied in the hydrologic model, using the description and formula provided in Section 1.6. If a large, complex pre-existing model is used for the study, the level of effort required to rename the reservoirs is not justified, and instead a lookup table shall be provided in .json form within the main watershed directory.

    The reservoir description field shall always include the authoritative NID Name field associated with the primary dam on the pool. The description shall also include any alternate, local, or familiar names used to reference the reservoir and dam.

    • examples: nid_tx05966

    Dam Names¶

    Dams are sub-elements of reservoirs, and their names will matter in watersheds that have reservoirs with diverted outlets or multiple dams. Name these sub-elements “dam”, or in cases with multiple dams at a reservoir, use append dam with a short descriptor, e.g., “dam-aux”. This descriptor is referred to in other sections of this document as ‘dam-outlet-descriptor’.

    Hydraulic Model Names¶

    Project File Name¶

    The modeling unit name should be based on the largest stream in the model unit and will commonly match a HUC name that may approximately be coincident with the model unit. The model unit will be defined during the Scoping and Data Preparation Phase.

    • formula: model-unit

    • example: lower-new

    Project Description Field¶

    All models created for FFRD will include a thorough description field providing users with pertinent information. The following items are listed in the HEC-RAS model description field with each item on a new line using a colon to separate the item name and response:

    • Program and Version: HEC-RAS 6.X.X September 20XX

    • Status of Model: Interim Review, Final Review, Final Draft, or Final

    • Primary Basin: Primary Basin Name

    • Modeling Unit: Modeling Unit Name

    • Date of last modification: MM/DD/YYY

    • Model Developed by: Company/Agency Name

    • Vertical datum and units: NAVD 88, feet

    • Horizontal datum and units: NAD 1983, feet

    • Horizontal Projection: USA_Contiguous_Albers_Equal_Area_Conic_USGS_version

    Geometry Name¶

    All models utilize a single geometry named using the modeling unit name (identical to the geometry).

    • formula: model-unit

    • example: lower-new

    Unsteady Flow Name¶

    The flow file for Calibration Phase modeling efforts will be named to match the project name, with a filename extension in the form of .u0X. The name of each unsteady flow event will be named according to the calibration event name of the event. The format for historic unsteady flow names for historic events is mmmYYYY. If historic events are increased with a multiplier for the purposes of a stress test the field is prefixed with a #x-mmmYYY format and considered a descriptive name.

    • formula: event-name

    • examples: apr1997, 2x-apr1997, sst

    Plan Name¶

    The plan file name for Calibration Phase modeling efforts will be named to match the project name, with a filename extension in the form of .p0X. The name of each plan will be named based on the scenario type and the calibration event name of the event. The plan file name for conformance/production phase models will be a fixed name, “sst”.

    Calibration Phase

    • formula: event-type _ event-name

    • examples:

    • calibration_apr1997

    • validation_jun1988

    • breach_apr1997

    • stress-test_2x-apr1997

      Conformance and Production Phase

    • formula: event-type (fixed)

    • sst

    Plan Short ID¶

    The short plan name must be identical to the plan name.

    Terrain Names¶

    The hydraulic model terrains are created using RASMapper software using the full resolution DEM created during the Scoping and Data Preparation Phase as the basis. The terrains will be named using the model unit and resolution as in the formula below. Note that the formula includes an optional field to indicate terrains that have been exported into a single raster and include no RASMapper based terrain modifications (i.e. ‘merged’). This name is the file name for the .hdf and .vrt files. No standard is set for the underlying .tif files that make up the terrain, but human readability should be considered.

    • Formula: model-unit _ ‘terrain’ _ resolution [ _ ‘merged’ ]

    • Examples:

    • little-fork-creek_terrain_1m

    • little-fork-creek_terrain_3m_merged

    Two-Dimensional Flow Area Element Names¶

    Two-dimensional (2D) flow area names will be identical to the project file name established in the naming convention table and used directly. Optionally, names may include an integer field, starting with 1, to distinguish when multiple areas are used in the model. If more than one 2D flow area exists in the model, the integer must be used for all elements.

    • formula: model-unit [ _ integer]

    • example: lower-new_1

    Boundary Condition Names¶

    Multiple distinct boundary condition line uses are called for in this SOP, and the naming formulas will vary for each based on the categories below.

    Inflow Boundary Condition from Upstream Basins¶

    The inflow boundary condition (BC) lines that supply inflow from upstream basins have names that descriptively indicate the link to an upstream basin and are intentionally repeated to be identical to the reference line from the upstream source. An optional integer field may be added so that multiple locations for passing flow between models can be distinguished when needed. The formula below will be followed.

    • formula: model-unit (upstream) _ ‘to’ _ model-unit (downstream) [ _ integer ]

    • examples:

    • mount-creek_to_lower-new

    • baldhill_to_sunny-valley_1

    Internal Boundary Condition from Subbassin Baseflows¶

    The inflow BC lines that are sourced from HEC-HMS subbasin base flows include an indicator that they are BC lines, the reference name, ‘s’ and unique ID from the HMS subbasin that supplies the flow.

    • formula: ‘bc’ _ reference-name _ ‘s’ unique-id _ ‘base’

    • example: bc_clear-crk_s230_base

    Internal Boundary Condition for Dam Outflows¶

    The flow hydrograph boundary condition (storage area/2D area connections) that are internal to the HEC-RAS model extent and are sourced from either observed flows, HEC-ResSim simulated outflows, or HEC-HMS simulated outflows are named according to the formula below.

    • formula: ‘nid’ _ nid-id _ [dam-outlet-descriptor]

    • example:

    • nid_tx05966

    • nid_ms02541_spill

    • nid_ms02541_gates

    Downstream Outflow Boundary Condition¶

    The outflow BC lines include a BC line indicator, the model unit name, an outflow indicator and an optional integer so that multiple outflows can be distinguished if necessary.

    • formula: ‘bc’ _ model-unit _ ‘out’ [ _ integer ]

    • example: bc_lower-new_out_1

    Excess Precipitation (Meteorological) Boundary Condition¶

    No name is applied to Meteorological Data boundary condition in HEC-RAS. In a desktop HEC-RAS model, the path to the source of the data is a simple .dss path link included in the plan file.

    Initial Condition Point Names¶

    Initial conditions points can be used to set the initial water surface elevations at the start of simulations and will commonly be associated with reservoir pools. When they are used to set an initial pool level, they will be named with an type identifier, a descriptive name for the reservoir and an optional integer. When initial conditions points are used for other purposes, no specific formula is required.

    • formula: ‘ic’ _ ‘nid’ _ nid-id [ _ reservoir-name ] [ _ integer]

    • examples:

    • ic_nid_ms58374

    • ic_nid_tx00776_homme-dam_1

    Storage Area/Two-Dimensional Connection Names¶

    Internal storage area connections are used to represent dams and levees as well as other unique features. The names of these structures will use and indicator of the type of structure followed by a numeric identifier according to the naming formulas and descriptions below. Note that a character limit of 32 applies for all storage area/2D connection names. Additionally, requirements for the description fields are also included for each type of structure below.

    Dams¶

    Dam names will directly correspond with the element names applied in the hydrologic and/or reservoir operations models, using the description and formula provided in Section 1.6.

    The reservoir description field will include the authoritative NID Name field.

    • examples: nid_tx05966, nid_mn00584_s002

    Levees¶

    2D connections representing levees will be named using the ‘nld’ followed by the National Levee Database System ID (System ID). The name will also include an integer field for cases when the levee system is broken into multiple structures (in order to allow multiple breaches or other reasons). When the integer field is used, it should be applied starting at 1 on the upstream and increasing downstream.

    The description field will include the authoritative name of the levee system found in the NLD (NLD System Name) on the first line, and a list of all NLD Segments IDs and NLD Segment Names that make up the system.

    • formula: ‘nld’ _ nld-system-id [ _ integer ]

    • examples: nld_3005000153, nld_505000356_2

    • Description Example:

    Prairie du Pont / Fish Lake System

    2 Segment System

    5604160001 - Fish Lake Drainage & Levee District No. 8

    5604350001 - Prairie du Pont Levee and Sanitary District

    Other Structures¶

    Other types of structures modeled using storage area/2D connections will be named using a ‘conn’ prefix, followed by a suitable ID, followed by a suitable descriptor of the structure type. The description field should be used to document the authoritative name of the structure and other useful information.

    • formula: ‘conn’ _ other-id _ other-type-description

    • examples: conn _1_bridge, conn _4_rr-embank, conn_3_ diversion, conn_10_roadway

    Reference Line Names¶

    Reference lines are used for several distinct purposes. Naming conventions for reference lines for each these purposes are detailed below.

    Observed Data Gage¶

    Reference lines that capture modeled output at observed stream gage locations must be named with an identifier indicating the purpose of the reference line (‘gage’), an indicator of the gage operator, and the gage operator’s ID. The gage operator ID will be sourced from the authoritative databases (commonly hosted by USGS or NOAA). Lists of the gages and complete observed data will be documented during the Scoping and Data Preparation Phase and provided to the model developer. Optionally, the gage name or an abbreviation can be appended to improve human readability.

    • formula: ‘gage’ _ gage-operator _ gage-id [ _ reference-name]

    • examples:

    • gage_usgs_12105900_gr-at-auburn

    • gage_usgs_12105900

    • gage_noaa_8594900

    • gage_noaa_8594900_washington-dc

    • gage_usace_1290183026_green-peter-outflow

    Hydrologic Model Elements¶

    Reference lines are included in the hydraulic model at all relevant hydrologic model elements such as headwaters subbasin outlets, junctions and reservoirs to allow for direct comparison of modeled results across each model. These reference lines will be named identically to the corresponding hydrologic model element.

    • Formula: see Subsections 1.6.4.1.1, 1.6.4.1.3, and 1.6.4.1.4.

    • Examples:

    • s-grn-50

    • white-rock-ck_s040

    • trinity-river_r040

    • west-fork_j090

    • west-fork_j090

    • elm-fork+clear-ck_j052

    • lewinsville-inflow_j099

    • hhd-reservoir-out_j010

    • trinity-river_r040

    • white-rock-ck_s040

    • nid_tx05966

    • nid_mn00584_s002

    Outflow Linking Locations¶

    Reference lines will be created to capture outflows from the models that will be passed to downstream models (note that outflow boundary condition lines will not be used for this purpose). The reference lines that pass flow to downstream model units will have names that descriptively indicate the link to the downstream model and are intentionally repeated to be identical to the inflow boundary condition name of the downstream model. An optional ‘integer’ field may be added so that multiple locations for passing flow between models can be distinguished when needed.

    • formula: model-unit (upstream) _ ‘to’ _ model-unit (downstream) [ _ integer ]

    • examples:

    • mount-creek_to_lower-new

    • baldhill_to_sunny-valley_1

    • baldhill_to_sunny-valley_2

    Other Reference Lines¶

    Reference lines for other purposes should use ‘ref-ln’ prefix followed by a location name.

    • formula: ‘ref-ln’ _ location-name

    • example: ref-ln_sunrise-bridge

    Reference Point Names¶

    Reference points are used in the model to capture water surface elevations at stream gages and loading conditions along levees. They also may be used for other miscellaneous reasons.

    Observed Data Gage¶

    Reference points will be included at each stream gage location and placed near the river centerline. Note that there is a 32-character limit for reference point names in HEC RAS. The ‘reference-name’ field is optional and may be left out if the character limit does now allow.

    • formula: ‘ref-pt’ _ ‘gage’ _ gage-operator _ gage-id [ _ reference-name]

    • examples:

    • ref-pt_gage_usgs_12105900_gr-at-auburn

    Levee Reference Points¶

    Reference points will be included along storage area/2D connections at levees located at the upstream end of the levee system, mid-point of the levee system, and downstream end of the levee system. Additionally, and two reference points will be included at each levee breach location (riverside/headwater and landside/tailwater). An integer value should be appended when more than one breach location is included for the levee system.

    • formula: ‘nld’ _ nld-system-id _ pt-purpose [ _ integer ]

    • examples:

    • nld_3005000153_us (Upstream)

    • nld_3005000153_ms (Midstream)

    • nld_3005000153_ds (Downstream)

    • nld_3005000153_br-hw (Breach headwater)

    • nld _3005000153_br-tw (Breach tailwater)

    • nld _5009000634_br-hw_2 (Breach headwater)

    Other Reference Points¶

    Reference points for other purposes should use ‘ref-pt’ prefix followed by a location name.

    • formula: ‘ref-pt’ _ location-name

    • example: ref-pt_sunrise-bridge

    The structure inventory file for the entire study basin will be delivered in .gpkg format. The naming convention will include hydraulic modeling unit basin identifier, NSI version, and the year of the price level, as shown in the formula below
