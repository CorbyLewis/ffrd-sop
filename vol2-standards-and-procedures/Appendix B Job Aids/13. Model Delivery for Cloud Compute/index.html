    Job Aid 13—Model Delivery for Cloud Compute¶

    Worflow Context¶

    This job aid describes the specific requirements and steps for delivery of models for cloud compute of synthetic events during the Conformance Phase. The model integration lead is responsible for delivery and verification of quality control (QC) checks of all models to the cloud as well as the associated final Model Linking Register that includes all inputs required for successful development of cloud compute manifests and execution in the cloud.

    The model integration lead is also responsible for several steps that must occur in the desktop environment to validate accurate linking of all models and to confirm model performance (which are not described in this job aid). The pre-conformance review checklist must be completed with all items closed to ensure the linked system of models has been validated and all cloud-preparation steps have been completed.

    Figure 1. Models are delivered for cloud-compute during the Conformance Phase of the project, the third phase of the FFRD Workflow.

    Figure 2. Focus of this Job Aid in relation to steps within the Conformance Phase.

    Final Model Linking Register Delivery¶

    A fully developed Name Table, Model Linking Register, Dam Scoping Table, and Levee Breach Locations are developed during the Scoping and Data Preparation Phase. These products provide the guide for model linking strategy throughout the FFRD project. They may be updated during the Calibration Phase or Conformance Phase of the project when unique issues are identified.

    The Model Linking Register is finalized early in the conformance phase when the full system of hydraulic models are linked in a desktop environment and tested. This Model Linking Register is an easily readable table that will contain all information required to build cloud-compute manifests described in detail in a separate job aid. These cloud-compute manifests control the actual linking of output data to input data that is summarized in the Model Linking Register with cloud-compute. Several cloud-compute manifests are required and include those for actions included in various plug-ins (i.e., hms-mutator-plugin, hms-runner, ressim-runner, fragility-curve-plugin, and ras-runner)

    Scripting to automate certain steps in the production of the compute-manifests has the potential to improve efficiency and ability to QC linking data.

    Directory Structure¶

    The desktop version of the model, prior to upload to the cloud should include internal referencing to input files as shown in the schematic below. This structure mimics the directory structure of how the output data from individual events are stored in Model Library by cloud-compute, which allows for efficient viewing and replication of cloud results in later steps.

    Note that the terrain is stored at a higher-level directory where all models can share it.

    The location for upload of the models for cloud-compute is not shown in this figure. The actual model upload locations are described separate from this document (unique for each model).

    Figure 3. Model Library Output Data Directory Structure to be Mimicked on Desktop Prior to Model Upload. Not the Location for Model Upload for Cloud-Compute

    Naming Convention Control¶

    The FFRD SOP includes robust specific criteria for naming conventions that apply to model file names and internal model element names. These naming conventions are critical for the ability to control model linkages, perform various QC steps, and reliably access output data among other uses.

    The most critical moment in the FFRD process to ensure naming conventions follow the standards is at model upload for cloud compute.

    Although the SOP includes various review steps and checklists to address naming convention issues, there is opportunity for human error. Automated processes to check for adherence to naming conventions using scripts has the potential to reliably catch issues and ensure consistency across the FFRD program.

    Time Window for Stochastic Storm Transposition Events¶

    The model integration lead will guide the event time window duration selection as described in the main SOP. It will be long enough to ensure rainfall in the upper basin has sufficient travel time to so that the downstream boundary of the model reaches a peak.

    A specific starting date and time will also be selected for use for the SST events and model preparation for cloud compute, but it will be varied for each event within cloud-compute.

    Hydrologic Modeling System Model Delivery¶

    The following sections serve as a checklist for transforming a calibrated HEC-HMS project and preparing it for Conformance and Production simulations using cloud-compute.

    The synthetic event sampling of initial states and basin parameter sets is based on the HEC-HMS modeler or model integration lead delivering a large dataset .basin and .control files that represent possible combinations of conditions and parameter sets on each day throughout the period of record used for stochastic storm transposition (SST).

    Software Version¶

    The HEC-HMS software version used for model development must be verified and consistent with the version called for in Section 6 of SOP Volume II.

    First Steps¶

    First, use the File>Save As... option within the HEC-HMS user interface to create a copy of the project. Use the Copy DSS data (time-series, paired, grid data) option to internalize any externally referenced HEC-DSS data as shown in the Figure 4. Note, this process does not copy external data like .asc or .tif files.

    Figure 4. HEC-HMS Save Current Project As Dialog

    Model Settings¶

    The following subsections describe modifications that must be made to HEC-HMS model files and project configurations to support stochastic simulations during the Conformance and Production phases.

    Basin Models¶

    Several modifications must be made to the basin models, which were created during the historic event calibration phase. In general, initial conditions for all basin elements (e.g., subbasins, reaches, and reservoirs) must be specified in absolute, not relative terms. This enables efficient stochastic sampling and specification of their values.

    Subbasins¶

    The Structured Discretization method must be used. The standard hydrologic grid (SHG) projection must be used along with an appropriate cell size (e.g., 2,000 meters), as shown in Figure 5.

    Figure 5. HEC-HMS Subbasin, Discretization Component Editor

    The Simple Canopy method must be used. The Depth Initial Storage Method must be used. All other parameter selections/values achieved during model calibration should be left unchanged. Figure 6 shows an example of these options.

    Figure 6. HEC-HMS Subbasin, Canopy Component Editor

    Note that the initial storage depth will be stochastically sampled and set during conformance and production simulations.

    If snow accumulation and melt processes are relevant to the watershed of interest, the Gridded Temperature Index Snow method must be used (otherwise, this process should be set to None). For each initial state variable, the Parameter Grid option must be used. All other parameter selections/values achieved during model calibration should be left unchanged. Figure 7 shows an example of these options.

    Figure 7. HEC-HMS Subbasin, Snow Component Editor

    Note that the parameter grids for each state variable will be stochastically sampled and set during conformance and production simulations.

    If surface storage processes are relevant to the watershed of interest, the Simple Surface method must be used (otherwise, this process should be set to None). The Depth Initial Storage Method must be used. All other parameter selections/values achieved during model calibration should be left unchanged. Figure 8 shows an example of these options.

    Note that the initial storage depth will be stochastically sampled and set during conformance and production simulations.

    The Deficit and Constant Loss method must be used. All parameter selections/values achieved during model calibration should be left unchanged. Figure 9 shows an example of these options.

    Note that the initial deficit will be stochastically sampled and set during conformance and production simulations.

    The ModClark Transform method must be used. All parameter selections/values achieved during model calibration should be left unchanged. Figure 10 shows an example of these options.

    Figure 8. HEC-HMS Subbasin, Surface Component Editor

    Figure 9. HEC-HMS Subbasin, Loss Component Editor

    Figure 10. HEC-HMS Subbasin, Transform Component Editor

    The Linear Reservoir Baseflow method must be used. The Discharge Initial Type must be used. All other parameter selections/values achieved during model calibration should be left unchanged. Figure 11 shows an example of these options.

    Figure 11. HEC-HMS Subbasin, Baseflow Component Editor

    Note that the GW1 and GW2 Initial discharge values will be stochastically sampled and set during conformance and production simulations.

    Reaches¶

    Depending upon the physical processes within the watershed of interest, multiple reach routing methods may be utilized within HEC-HMS FFRD applications. Regardless of the chosen routing method, the Specified Discharge Initial Type must be used. All other parameter selections/values achieved during model calibration should be left unchanged. Figure 12 shows an example of these options.

    Figure 12. HEC-HMS Routing Component Editor

    Note that the initial discharge values will be stochastically sampled and set during conformance and production simulations.

    Reservoirs¶

    Depending upon the physical processes within the watershed of interest, multiple reservoir routing methods may be utilized within HEC-HMS FFRD applications. Regardless of the chosen routing method, the Elevation Initial Condition must be used. All other parameter selections/values achieved during model calibration should be left unchanged. Figure 12 shows an example of these options.

    Figure 13. HEC-HMS Reservoir Component Editor

    Note that the initial elevation values will be stochastically sampled and set during conformance and production simulations.

    Meteorologic Model¶

    Unlike the basin models, a brand-new meteorologic model named, SST, should be created for use during the Conformance and Production phases. The Metric Unit System must be specified (this is used when transposing the gridded boundary conditions). For all FFRD applications, the Gridded Precipitation method must be selected. If snow accumulation/melt or evapotranspiration processes are relevant to the watershed of interest, the Gridded Temperature method must be selected (otherwise, this process should be set to None). Similarly, if evapotranspiration processes are relevant to the watershed of interest, the Gridded Hamon Evapotranspiration method must be selected (otherwise, this process should be set to None). The Replace Missing option should be specified as Set to Default. Figure 14 shows an example of these options.

    Figure 14. HEC-HMS Meteorology Model Component Editor

    Within the Gridded Precipitation node, any valid precipitation grid name can be chosen. These datasets will be stochastically sampled and set during conformance and production simulations. The Normalize Start Time Shift Method must be selected. The Transpose option should be set to Yes. Default values near the middle of the watershed in the basin model's coordinate reference system should be specified within the X and Y Coordinate entry fields. Figure 15 shows an example of these options.

    Figure 15. HEC-HMS Meteorology Model, Gridded Precipitation Component Editor

    If the Gridded Temperature method was selected, within the Gridded Temperature node, any valid temperature grid name can be chosen. These datasets will be stochastically sampled and set during conformance and production simulations. The Normalize Start Time Shift Method must be selected. Figure 16 shows an example of these options.

    Figure 16. HEC-HMS Meteorology Model, Gridded Temperature Component Editor

    If the Gridded Hamon method was selected, all parameter selections/values achieved during model calibration should be left unchanged within the Gridded Hamon node. Figure 17 shows an example of these options.

    Figure 17. HEC-HMS Meteorology Model, Gridded Hamon Component Editor

    Control Specifications¶

    Similar to the meteorologic model, a brand-new Control Specification model, named SST, should be created for use during the Conformance and Production phases. 01Jan2100 should be entered within the Start Date entry field. 00:00 should be entered within the Start Time entry field. 04Jan2100 should be entered within the End Date entry field. 00:00 should be entered within the End Time entry field. Depending upon the physical processes within the watershed of interest, multiple computational time steps may be utilized within HEC-HMS FFRD applications. Figure 18 shows an example of these options.

    Figure 18. HEC-HMS Control Specifications Component Editor

    Simulation Runs¶

    Similar to the control specifications, a brand-new simulation run model, named SST, should be created for use during the Conformance and Production phases. Any basin model can be selected within this simulation as they will be updated during the conformance and production simulations. The SST meteorologic model and SST control specifications (which were previously created) should be selected. The Spatial Results option should be set to Yes and the Spatial Interval should be set to 1 Hour. Figure 19 shows an example of these options.

    Figure 19. HEC-HMS Simulation Run Component Editor

    The No Ratio method should be selected on the Ratio tab. Additionally, the No Save States option should be selected on the States tab.

    Importing Precipitation Data (Storm Catalog)¶

    The tool used for generating storm catalogs produces a .grid file that follows the specification of a normal HEC-HMS .grid file that links gridded datasets to a project. This lets you quickly copy and paste a list of entries for hundreds of storms into your project and effectively import them without needing to manually create hundreds of gridsets in HEC-HMS.

    For the steps below, the project.grid file is the .grid file that exists in the HEC-HMS project and matches its name. It contains all the grid data used in calibration and POR simulations The catalog.grid file is the one that was created by StormCloud/StormHub and only contains the precipitation data for storms in the storm catalog.

    The instructions below assume you are using Notepad++ to edit the files.

    Make sure your HEC-HMS project is closed before editing any internal files.

    1. Copy (or move) the storm data .dss files to the project data folder in a subdirectory called catalog.

    2. Make a backup copy of the catalog.grid file and give it the file extension .gridbak with the same filename as the original catalog.grid file.

    3. Open the original (not backup) catalog.grid file in a plaintext editor.

    4. Find and replace file paths to point to the project's catalog folder.

    1. Scroll down to the first grid data entry in the catalog.grid file and find the DSS File Name entry.

    2. Highlight the portion of the file path up to the last backslash (just before the file name), as shown in Figure 20.

    Figure 20. Section of Text to Replace within .grid File

    1. Press Ctrl+H to open the Replace dialog. Note that it automatically places the highlighted text in the Find what field.

    2. In the Replace with field, enter data\catalog.

    3. Click Replace All on the right side.

    1. Save the catalog.grid file but leave it open.

    2. Make a backup copy of the project.grid file and give it the file extension .gridbak with the same filename as the original project.grid file.

    3. Open the original (not backup) project.grid file in a plaintext editor in a new window.

    4. Scroll to the bottom of the project.grid file and find the last grid entry. Add a blank line after the "End:" tag for the last grid, and then press Enter again to go to the following line.

    1. Each entry must be separated by a blank line.
    1. From the catalog.grid file, copy all the storm grid entries and paste them into the project.grid file below the blank line you added.
    1. Make sure not to copy the catalog.grid file's header (the first four lines beginning with "Grid Manager: ...”)
    1. Save the project.grid file. Close both .grid files.

    When you re-open your HEC-HMS project, you will see that hundreds of new gridsets are listed in the Watershed Explorer.

    Model Cleanup¶

    The HEC-HMS project used during the Conformance and Production phases should be as lean as possible because it will be copied many times and reproducing unnecessary data is a waste. Below are several things that help reduce the size of the SST project. Most of the cleanup can happen within the HEC-HMS user interface (e.g., deleting basin models) but there are hints below where deleting a large number of entries using the HEC-HMS input files is faster (i.e., within Notepad++). Always be careful when editing HEC-HMS input files and make a backup before doing so.

    Basin Models¶

    First, remove any basin models except for the finalized event calibration basin models. Also, remove any linkages to observed data (e.g., flow, stage, SWE) on the Options tab (set to none) of all basin elements (e.g., subbasins, junctions, reaches, etc.) as this information will not be utilized during stochastic simulations.

    Meteorologic Models¶

    Next, remove any met models except for the SST met model you created in the steps above.

    Control Specifications¶

    Also, remove control specifications except for SST from the project.

    Time-Series Data¶

    It is unlikely that you will need any time-series data if you have unlinked all your observed data, are using gridded processes, and do not have any specified discharge elements (reservoirs, sources, etc.) in the model. If you are confident that you need certain time-series data, do not delete them.

    The following process removes the links from the HEC-HMS project, but you will need to clean up the DSS files manually. The DSS files that contain the time-series data were copied over during the initial Save As step. You can use the File>References tool and filter to Time Series Data to see where these data are on disk and delete them manually. They should have been internalized to the copied project's data folder and are safe to delete because they are copies of the original.

    1. Make a backup copy of the project.gage file that exists in your HEC-HMS project and matches its name. Give it the file extension .gagebak with the same filename as the original project.gage file.

    1. Open the original (not backup) project.gage in a plaintext editor.

    2. Delete everything below the file header (first 5 lines).

    Paired Data¶

    Unlike time-series data, it is very likely that several paired data objects are utilized within the basin models that will be used during the stochastic simulations. Thus, care needs to be taken to organize these paired data objects in such a way that any errors or compute issues can be efficiently investigated during the conformance and production phases.

    Begin by culling any unused paired data objects (i.e., storage-discharge, elevation-storage, etc.) within the HEC-HMS project. Then, for the remaining paired data references that utilize external DSS files, combine all records into a single DSS file. This paired data DSS file should be placed within the project's data folder. For any paired data references that utilize the Manual Data Entry method, these records can remain unaltered.

    Simulation Runs¶

    Remove any simulation runs or any other computes (e.g., Uncertainty Analysis, Optimization Trial, etc.) from the project except for the SST run you created. Other compute types can be removed by deleting directory within the base project directory (with the HEC-HMS project closed):

    • Delete the montecarlo directory to delete all Uncertainty Analyses.

    • Delete the optimizer directory to delete all Optimization Trials.

    • Delete the ensemble directory to delete all Ensemble Analyses.

    • Delete the forecast directory to delete all Forecast Alternatives.

    • Delete the frequency directory to delete all Frequency Analyses.

    DSS Files in the Project Directory¶

    There will be some straggler files associated with now-deleted simulations to clean up. Delete any DSS files with a name that matches a meteorologic model which was removed. These files contain a met data cache and can be removed. You can verify this is the case by opening the DSS file in DSSVue and verifying:

    • It only contains meteorologic data.

    • The F-part for all paths contains "MET:xxxxx" where "xxxxx" is the name of the met model.

    Delete any DSS files with a name that matches a simulation run which was removed. These files contain a They can be deleted. You can verify this is the case by opening the DSS file in DSSVue and verifying:

    • It only contains simulation results.

    • The F-part for all paths contains "RUN:xxxxx" where "xxxxx" is the name of the simulation run.

    Delete any DSS files with a name that matches other simulation types (e.g., Uncertainty Analysis, Optimization Trial, etc.) can be deleted as well.

    Other Files in the Project Directory¶

    All files with the .out and .log extensions can be safely deleted. Additionally, all .sqlite files that do not match one of the basin models that remain within the HEC-HMS project can be deleted. Finally, the results directory can be safely deleted.

    Cleaning Up GIS Data¶

    There will not be modification of the geospatial information for any basin element, so the source or processed terrain data will no longer be necessary. Thus, the gis, maps, and terrain directories should be deleted.

    Developing Continuous Initial Snow States¶

    This process is only required for projects that utilize the Gridded Temperature Index snow method:

    1. First, re-run the period of record with full spatial results turned on to get full set of gridded state variables for snow.
    1. Second, sanitize the output gridded data sets using the Vortex | Sanitizer tool.

    HEC-HMS Model Delivery¶

    The prepared and cleaned up model files must be delivered by posting them to the appropriate location on the project directory. There are two distinct deliveries associated with the HEC-HMS model delivery: the runnable model and template files for use by cloud-compute tools to generate input files that account for variability in antecedent conditions for each stochastic event.

    Base Model¶

    Post the prepared and runnable HEC-HMS models to the following location within the project directory:

    Delivery Location: conformance > hydrology > [basin-name]

    Figure 21. Example Delivered HMS Model Dataset

    Antecedent Condition Template Files¶

    The basin-record-gen-plugin cloud-compute actions take HMS template files to create an array of .basin and .control files to represent the variation in antecedent conditions for stochastic events. The input template files to be delivered are:

    • Calibrated Period of Record HECHMS DSS output

    • Individual .basin files from each calibration event

    • Individual .control files from each calibration event

    The template files that will be used by cloud compute will be posted to the location below.

    Delivery Location: conformance > basin-generation

    HEC-ResSim Delivery¶

    The following sections serve to describe how to prepare and deliver the HEC-ResSim FFRD model for conformance phase and production phase runs using Cloud Compute.

    Software Version¶

    The HEC-ResSim version used for model development must be verified and consistent with the version called for in Section 7 of SOP Volume II.

    Create the Conformance Run Template¶

    A single alternative and a single simulation shall be the starting point for the conformance model:

    • Conformance alternative name=“fema_ffrd”

    • Conformance simulation name=“fema_ffrd”

    Conformance Alternative¶

    The simplest way to create the “fema_ffrd” alternative is to make a copy of one of the existing calibration alternatives.

    1. From the Reservoir Network Module, select the Alternative Editor, click on one of the calibration runs, and use the Save As… option.
    1. Save as “fema_ffrd” and provide an informative Description such as “FEMA FFRD Cloud Compute Alternative – Originated as a copy of April 1990 calibration alternative for use with ffrd cloud computes”

    Figure 22. Step 2 Window

    1. No other changes should be necessary but ensure that the appropriate timestep and flow computation method are selected. The inflow mapping must be consistent with the simulation date range and timestep.

    Conformance Simulation¶

    Next move to the Simulation Module and create the new “fema_ffrd” simulation.

    1. As previously state, it is easiest to start with the calibration simulation that was paired with the alternative used to create the “fema_ffrd” alternative, because the simulation dates must match the available data. First open the matching calibration simulation, and use the right-click secret menu to copy the Simulation Time Window.

    Figure 23. Step 1 Window

    1. Use the Simulation>Create a New Simulation option to create the fema_ffrd simulation, and give it an informative description such as, “FEMA Future of Flood Risk Data (FFRD) Cloud Compute Simulation.”

    2. Use the secret right-click Paste TimeWindow option to make the simulation time window match the currently available data. The cloud compute modeler will automate the updates to these dates to match what is sampled during the production runs.

    3. Run the simulation to ensure that nothing has broken, and note the total compute time for the report.

    Figure 24. Step 4 Window

    Within Model Cleanup¶

    As with HEC-HMS, the HEC-ResSim watershed used during the Conformance and Production phases should be as lean as possible because it will be copied many times and reproducing unnecessary data is a waste. While some model-cleanup was done during the calibration modeling, the following steps shall be taken to ensure the most compact, streamlined model for conformance and production.

    Note

    Always be careful when directly editing HEC-ResSim model files and make a backup before doing so.

    These instructions assume you have already done some cleanup and thorough documentation during earlier phases, e.g., removed all unnecessary networks and scripts, and any particularly messy, disorganized pieces of the model.

    Create a Zipped Backup Model¶

    Before zipping up your model, make sure the current state has been fully documented. Use the File>Watershed Info… feature to log a comment about the current state of the model.

    Figure 25. Zipped Backup Model Window

    Then completely close the model and all associated files before creating a zipfile back up. Now that there is a backup of the model, delete as much superfluous files as possible.

    Remove All Simulations except “fema_ffrd”¶

    Reopen the model and go to the Simulation Module. Delete all simulations except fema_ffrd, using the Simulation>Delete Simulation option.

    Figure 26. Delete Simulation Window

    Delete all Alternatives except fema_ffrd¶

    From the Network Module, Alternative Editor, deleted all but the fema_ffrd alternative. There is no bulk delete option, but Ctrl+D can be used to make this process slightly faster.

    Remove all Unnecessary Outputs in fema_ffrd alternative¶

    Generally, only simulated flow and elevation outputs are required to be saved. The outputs that are required for FFRD simulations either feed directly to a HEC-RAS model, aid in identification interesting events (event filtering), or provide the required data to develop a relevant frequency curve for conformance metrics.

    The DSS Output tab in the alternative writes all computed and interpolated time series by default. Prior to handoff, the user should uncheck both of these in the Alternative Editor. The user should also go a step further and uncheck additional items on the subtabs. On the Reservoir tab, the user should only need the Reservoir Pool and flows so all other outputs can be unchecked. The user can also uncheck all outputs on the Reaches and Diversion tabs. For Junctions and State Variables, the user may want to uncheck some outputs; however, it is recommended that any record that may be referenced during validation (such as downstream control points or data handoff points between models) should remain selected.

    Figure 27. Alternative Editor Window

    Excess Rules and Operation Sets¶

    Rules and operation sets create very little extra bulk to model file size, but can create clutter and confusion. To alleviate this, during the Calibration Phase, ensure that the intention behind rules was included in their descriptions, and that the operation sets used for FFRD runs were clearly marked. Cleaning up unneeded operation sets from a model with 16 reservoirs, each with 3 operation sets required 208 clicks but did not result in any noticeable file size reduction, so the modeler should focus on tidiness in this case, and may leave extra operation sets in the model without causing harm.

    ResSim Script Considerations¶

    ResSim models with scripts may be flagged as a security risk within the Civil Works Business Intelligence (CWBI) cloud, and no jars or python modules will be accepted on CWBI. To address this concern, modelers will clean existing models of scripts by reviewing each script function and replacing it wherever possible with simpler operations. If scripted operations (rules or state variables) are necessary, the modeler will conduct a careful review and write justification for the CWBI security team to support the need for the script’s inclusion on CWBI. Modelers should keep in mind that scripted rules and state variables will not be calculated during the HEC-ResSim lookback period, and this can impact overall model results. Scripts must be structured to only require the data available within the event.

    HEC-RAS Delivery¶

    A final calibration phase HEC-RAS model for specific historic storm events is the starting point for the Conformance Phase.

    Various changes are made to prepare the model for generalized synthetic storm events driven by SST before they are posted to the Model Library in preparation for cloud-compute, which are described in the sections below.

    Software Version¶

    The HEC-RAS software version used for model development must be verified and consistent with the version stated in Section 8 of SOP Volume II.

    Plans and Naming Conventions¶

    HEC-RAS model is delivered with a single plan file and single flow file with generalized naming conventions to accept SST events that will not be confused for actual observed events. The following changes are made to the original calibrated model names:

    • Project and Filenames: [model-unit]

    • Geometry Name: [model-unit]

    • Plan Name: SST

    • Plan Short Name: SST

    Plan File¶

    The following adjustments should be made on the desktop version of the model plan file.

    Time Window¶

    The simulation time window selected for SST events, described section 1.4 above, will be used for the generalized SST event HEC-RAS time window. It will match the time window used in HEC-HMS and HEC-ResSim.

    Note that the time window is somewhat imperfect in that initial conditions in the HEC-RAS model start during the look-back period prior to the 72-hour precipitation event. The processes for FFRD SST event time windows, start date, and initial conditions will be considered for improvement in future versions of the SOP.

    Warm-up Time Period¶

    The plan file warm-up time period settings should be set to balance accuracy and efficiency in terms of cost and computation time in the cloud-compute environment.

    The use of a warm-up time period allows for the channels in the model to be primed with water before the start of the simulation. Note that in the context of the FFRD HEC-RAS models, excess precipitation gids will not contribute to initial flows in channels so the flow boundary conditions will be the primary contributor during the warm-up, if used.

    In some cases, the modeler may determine that warm up time periods do not improve the ability of the HEC-RAS model to predict peak water surface elevations, and it may not be needed.

    Figure 30. HEC-RAS Unsteady Computation Options and Tolerances Window

    Initial Conditions Points¶

    Ensure that the Keep initial elevations constant during warmup option is set to yes, which allows initial conditions (IC) points representing pool elevations that will be varied in SST events to remain constant when a warmup time period is also sued.

    Restart Files¶

    If restart files were used for the calibration runs, they must be removed for the cloud-based SST events.

    Breaches¶

    All breach locations with various breach parameters will be included in the final calibration phase model prior to the start of the Conformance Phase. For the model upload for cloud compute, all breaches should be turned on (Breach This Structure checkbox). The trigger breach elevation may be set to the toe of the structure to be breached, but cloud-compute will handle random sampling of this value during computations.

    Equation Set¶

    The 2D equation set selected in the plan file should be consistent with the selected equation set used for the calibration of the model. This should be verified prior to model upload for cloud compute.

    Matrix Solver¶

    The plan file 2D matrix solver selected in HEC-RAS may be evaluated to balance computational cost and numerical stability prior to delivery to cloud compute.

    Solver Cores Setting¶

    The plan file, Solver Cores setting on the run should be set to optimize efficiency in terms of cost and computation time in the cloud-compute environment. Coordination with the cloud environment support team is needed to determine the optimal Solver Core setting, which accounts for type of compute resource to be used and computation time. Note that the ras-runner compute manifest should also call the appropriate compute resources that match this HEC-RAS solver core setting.

    Figure 31. View of Window that Defines Solver Core Settings

    Output Control Options¶

    Ensure that no unnecessary output is written due to settings in the Output Control Options. It is common to turn on certain variables to be saved to the HDF5 output file during the model development and calibration process, but those should not be carried forward for the cloud-compute runs during the Conformance or Production Phases of the project.

    Figure 32. Example Setting for Output Control Options

    Mapping Output Interval¶

    The mapping output interval can have a significant impact on HDF output file size. The hydraulic modeler should coordinate with the model integration lead and other team members to investigate whether the critical event-level outputs can be achieved with a longer mapping output interval. Note that the maximum water surface elevation and maximum velocity for each cell is recorded at the computation interval, which lessens the need for a short mapping output interval.

    Geometry¶

    Although no special changes to the geometry file will be necessary to deliver the desktop model for cloud compute, it is possible that minor geometry adjustments will be needed in the conformance phase models due to unique issues that were not discovered during the calibration of individual models. Note if any updates to the models that affect linking take place, corresponding updates to Model Linking Register are needed. The model integration lead will own the Model Linking Register throughout the process.

    The geometry file (.g01) should be uploaded with the model to the compute directory. Note that the file is not required for cloud-compute, but it is retained to allow for a desktop modeler to work with the model locally in future steps if needed.

    Flow File¶

    Several updates to the flow file are needed prior to upload for cloud compute to fully link to the input data sources for SST events. Relative path name references should be based on the directory structure.

    Note that cloud-compute ingests only the HDF files for most of the key inputs for the computation and does not use the flow file. However, the flow file itself (.f01) should be uploaded with the model to the compute directory to allow for a desktop modeler to work with the model locally in future steps if needed.

    Observed Data¶

    Deleted references to observed data hydrographs from the calibration events. Rating Curve tables may be maintained.

    Upstream Inflows¶

    Any upstream inflow hydrographs must be updated to reference appropriate simulated outputs based on the Model Linking Register.

    Meteorological Data¶

    Excess Precipitation grids must be updated to reference HMS results from SST events.

    Reference the directory structure for relative pathnames for reference to the excess precipitation grids.

    Filename: exported-precip.dss

    HMS Baseflows¶

    Baseflow hydrographs must be updated to reference HMS results from SST events.

    Reference the directory structure for relative pathnames for reference to the baseflows.

    Filename: SST.dss

    Downstream Water Surface¶

    Appropriate adjustments to the downstream water surface boundary condition must be made. This may be an adjustment from an observed time series to a rating curve, or from an observed tidal stage time series to a representative fixed water surface during synthetic rainfall events. The main body of the SOP provides general guidance on selection.

    Dam Releases from ResSim or HMS¶

    Dam outflow hydrographs must be updated to reference ResSim or HMS results from SST events according to the Model Linking Register and Dam Model Scoping Table.

    Reference the directory structure for relative pathnames for reference to the simulated dam outflows.

    ResSim output filename: simulation.dss

    HMS output filename: SST.dss

    RAS Mapper File¶

    The RAS Mapper file must associate with the terrain-modeling file according to the directory structure. Reference to the default location for land cover should remain in place to allow for modelers to download the model and recreate results. Links to plans and results from calibration efforts or any data other than the single delivered plan must be deleted.

    The RAS Mapper file is also used in cloud-compute during depth and velocity grid generation. This process will refer to the separate path to a lower resolution grid. The requirements for this version of the RAS Mapper are not final as of June 2025.

    Run Plan on Desktop¶

    The HEC-RAS model plan must be executed on desktop to build the intermediate files that are used directly by cloud compute.     List Cells Affected by Initial Conditions Points¶

    The cloud compute will vary initial reservoir elevations for each event for all reservoirs modeled as reservoirs in HEC-HMS or included in the HEC-ResSim model, as defined in the Dam Scoping Table. Each of these reservoirs will include a HEC-RAS Initial Conditions point.

    The control for the varied IC point elevation is not within the flow file that desktop HEC-RAS users are familiar with. Rather, the reservoir pool starting condition are fed to the Linux/cloud compute for HEC-RAS through the .p01.HDF file, from the Event Conditions/Unsteady/Initial Conditions/2D Flow Areas/Perimeter 1 path.

    Figure 36. HDF File Path that Controls IC Point Starting Elevations

    This path includes an array of IC elevations for each cell in the model, and must be over-written prior to cloud compute. The HEC-RAS modeler integrator must deliver a list of all cells that could potentially be wet for the highest possible sampled IC to the cloud compute team. To do this, a temporary model run using a very-high-IC-point, high water level for the reservoir should be computed. After the computation completes, the resulting HDF file must be opened, the previously mentioned path tabulated and exported to a table using the HDFViewer software. This list of cells will be used by cloud compute for over-writing of pool levels in SST events.

    Note

    HEC-RAS will ignore an IC in cells with bottom elevations that are lower than the pre-set IC elevation. This is why it is appropriate to set IC elevations for all cells that are possibly wet for IC.

    1. Table > Select All > Copy. This data is the initial elevation applied to each cell in order of cell number.

    2. To reformat this data into a list of cell numbers. Paste these values into Excel. Insert a column to the left named Cell Num, number this column starting at 0.

    Create Exported Terrain for Mapping¶

    Merging and resampling of terrain to produce the terrain-mapping version is performed within HEC-RAS Mapper. This software provides adequate tools to perform this process seamlessly and minimizes opportunities for user error by using a consistent system. The merged/resampled output DEM should be generated directly from the HEC-RAS terrain which is associated with the HEC-RAS geometry file used for FFRD modeling. This terrain should include any vector terrain modifications used to alter the terrain, such as: hydro-connectors, culverts, bathymetry estimates, levees, etc.

    To merge and resample a terrain in RAS Mapper, right-click the terrain associated with the FFRD model geometry. Select Export Layer, and Export Raster as shown in Figure 40. The exported raster name should be descriptive and follow naming convention guidance as shown in Job Aid 06, Standard Naming Conventions. It should include the exported cell size as specified in Section 4 of SOP Volume II. The checkbox to Export To Single Raster should be enabled. This is required if the input HEC-RAS terrain .hdf file references several individual raster DEMs which may be of variable cell resolutions. The output cell size should be defined in feet.

    Note that terrain modifications should be incorporated into the exported merged/resampled terrain. By default, if vector terrain modifications are included in the HEC-RAS terrain .hdf file, they will be burned into the exported terrain. If the width of a terrain modification is small (i.e., substantially less than the resolution of the exported raster file), the terrain modifications may be misrepresented in the output terrain surface as smoothing can occur when rasters are resampled. This is generally acceptable for cloud result mapping but is a reason to beware when using resampled terrains for hydraulic modeling.
