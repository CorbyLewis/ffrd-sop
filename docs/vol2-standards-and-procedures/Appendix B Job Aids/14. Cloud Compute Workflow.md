### *Workflow Context*

*Compute manifests are the critical files that contain instructions for a cloud compute plugin to perform a compute job. They are \*.json formatted files that control many aspects of the cloud compute such as connections between inputs/outputs, paths for output storage, docker-file source (and version), and certain model parameters.*

*The model integration lead is responsible for delivering models and the final Model Linking Register for cloud compute as described in Job Aid 11 – Delivery of Models for Cloud Compute. This Model Linking Register is an easily readable table that will contain all information required to build many of the required cloud-compute manifests*

*The cloud environment lead is responsible for creation of compute manifests and execution of the cloud compute jobs. The purpose for each model manifest for FFRD cloud compute is provided in the table below, and more detailed descriptions are available through the cloud compute documentation (https://github.com/USACE-Cloud-Compute/cc-home). The manifests utilized for cloud compute throughout a project will be stored in the ‘conformance/manifests’ and ‘production/manifests’ directories.*

*Adhering to proper naming conventions and guidance for where and how to store models significantly improves the success of model integration and sets the stage for significant automation. This SOP provides strict guidance on naming conventions of directory structure, models, and model elements that facilitates the integration of the models.*

*Scripting to automate certain steps in the production of the compute-manifests has the potential to improve efficiency and ability to QC linking data.*

### *Conformance Phase Model Manifests and Computes*

*Utilizing the Model Linking Register and the models delivered for cloud compute, the cloud compute lead creates a set of model manifests constructed to support cloud computation. In addition to the delivered models, some other models supporting cloud compute are created.*

***<u>Cloud-Compute Setup</u>***

1.  *Create an event generator manifest on the local desktop. Coordinate with the meteorologists to confirm the target number of events per year.*

    - *Create a seed generator manifest and a seeds.json file that is uploaded to the checkpoint-validation/seeds folder within the watershed repository in the model library. The seeds.json file contains seeds for SST, HEC-HMS, fragility curves, and consequences are the standard seeds and likely the only seeds to consider. Verify with cadre lead that no other sampling is necessary.*

2.  *Upload the storm catalog, including listed files, to the conformance/storm-catalog folder within the watershed repository in the model library.*

    - *Catalog.grid*

    - *Transposition domain.gpkg*

    - *Watershed boundary.gpkg*

    - *DSS folder containing data for all 440 storms.*

3.  *Develop antecedent data sets utilizing the POR HEC-HMS run, the calibration event HEC-HMS runs, and the storm catalog. Post the antecedent data sets to conformance/hydrology/basinmodels.*

4.  *Develop a compute manifest for SST. Ensure the correct seed set is connected to the storm selection process.*

    ***<u>Initial Cloud-Computation</u>***

5.  ***System Response**: Develop a compute manifest for system response curves. Coordinate with Model Integration Lead to resolve any issues.*

    1.  *Run event \#1 through system response and verify the output is appropriate.*

    2.  *Run a full realization through system response curves. Identify any issues and adjust or work with the levee data team to adjust the system response curves as needed to conform to the expected trigger elevation distribution at the breach locations.*

6.  ***HMS**: Develop a compute manifest for HEC-HMS. Coordinate with Model Integration Lead to resolve any issues.*

    1.  *Run event \#1 through HEC-HMS.*

    2.  *Run event one through the integrated model on the desktop.*

    3.  *Model Integration Lead is responsible for verification that event \#1 cloud results are compatible with a replicated run in the desktop environment. Identify any issues and adjust or work with the HEC-HMS modeling team to adjust, the HEC-HMS model as needed.*

    4.  *Run a full realization through HEC-HMS. Identify any issues and adjust or work with the HEC-HMS modeling team to adjust the HEC-HMS model as needed ensure initial states and basin parameters are being properly sampled.*

    5.  *~~Run all calibration events using the manifests for the HEC-HMS model used for frequency calibration to validate the changes to the model to meet frequency calibration do not cause a model’s historic event calibration to fall out of acceptable ranges.~~*

7.  ***ResSim**: Develop compute manifest for ResSim utilizing the delivered model and the Model Linking Register. Coordinate with Model Integration Lead to resolve any issues.*

    1.  *Run event \#1 through HEC-ResSim.*

    2.  *Run event \#1 through the integrated model on the desktop.*

    3.  *Model Integration Lead is responsible for verification that event \#1 cloud results are compatible with a replicated run in the desktop environment. Identify any issues and adjust or work with the HEC-ResSim modeling team to identify and correct any issues.*

    4.  *Run a full realization through HEC-ResSim. Identify any issues and adjust or work with the HEC-ResSim modeling team to adjust the HEC-ResSim model as needed to ensure model is producing expected results.*

    5.  *~~Run all calibration events using the manifests for the HEC-ResSim model used for frequency calibration to validate the changes to the model to meet frequency calibration do not cause a model’s historic event calibration to fall out of acceptable ranges.~~*

8.  ***RAS**: Develop compute manifests for all RAS models utilizing the delivered models and the Model Linking Register. Coordinate with Model Integration Lead to resolve any issues.*

    1.  *Run event \#1 through HEC-RAS.*

    2.  *Run event \#1 through the integrated model on the desktop.*

    3.  *Model Integration Lead is responsible for verification that event \#1 cloud results are compatible with a replicated run in the desktop environment. Identify any issues and adjust or work with the HEC-RAS modeling team to identify and correct any issues.*

        ***<u>Fidelity Check</u>***

9.  *Run event 3 events through HMS, ResSim, System Response and RAS.*

10. *Run replicas the 3 cloud-computed events through the full integrated watershed model on the desktop (HMS, ResSim and RAS).*

11. *Modeling Integration Lead is responsible for verification that results are compatible. Model Integration Lead and Cloud Compute Lead will coordinate with any modeling team members necessary to identify and correct any issues.*

    ***<u>Single Realization Computation</u>***

12. *Develop a list of events that contribute to the annual maximum series based on HMS, ResSim and System Response results.*

13. *Run a full realization through HEC-RAS for frequency calibration, excluding events that do not contribute to the annual maximum series.*

    ***<u>Conformance Metric Evaluation</u>***

14. *The model integration lead, model team members, levee data lead, and cloud compute lead will evaluate H&H models against observed data at the primary, secondary, and tertiary gages. The conformance metrics will be used to guide acceptability of the results.*

15. *Adjust watershed model parameters as needed to achieve conformance to observed data*

16. *~~Run all calibration events using the manifests for the HEC-RAS model used for frequency calibration to validate the changes to the model to meet frequency calibration do not cause a model’s historic event calibration to fall out of acceptable ranges.~~*

    ***<u>Hazard Maps and Consequences</u>***

17. *Hazard Grids (events): Develop compute manifest for hazard grid generation utilizing the delivered models and the Model Linking Register. Coordinate with Model Integration Lead to resolve any issues.*

    1.  *Run hazard map production for event one.*

    2.  *Verify resulting hazard grid matches expected results for the same event with comparison in RASMapper. Coordinate with the Model Integration Lead to resolve any issues.*

    3.  *Compute hazard grids for all events with RAS results*

18. *Hazard Grids (aggregated): Develop compute manifest for aggregated hazard grids.*

    1.  *Run map aggregation for all results*

19. *Consequences: Develop compute manifests for consequences computation for each model unit.*

    1.  *Run a full realization through the consequences model. Identify any issues and work with the consequences modeling team to adjust as necessary.*

        ***<u>Conformance Metric Evaluation (re-visit)</u>***

20. *The consequence lead, model integration lead, levee data lead, model team members and cloud compute lead will evaluate H&H models results & consequence results and make any adjustments needed. The conformance metrics will be used to guide acceptability of the results.*

    ***<u>Conformance Phase Delivery</u>***

21. *Cloud compute team member computes all final summary outputs.*

22. *Conformance Review Checklist is finalized with all items closed out to document the review process throughout the phase.*

23. *Cloud compute team member computes average cost per event for cloud compute runs based on compute costs during conformance phase. Coordinate to ensure sufficient budget for production.*

### *Production Phase Manifests and Computes*

1.  *Copy the contents of the conformance/storm-catalog folder to the production/storm-catalog folder.*

2.  *Copy the contents of the conformance/hydrology folder to the production/hydrology folder. Only copy the final model; do not copy the archived models.*

3.  *Copy the contents of the conformance/reservoir-operations folder to the production/reservoir-operations folder. Only copy the final model; do not copy the archived models.*

4.  *Copy the contents of the conformance/hydraulics folder to the production/hydraulics folder. Only copy the final models; do not copy the archived models.*

5.  *Copy the contents of the conformance/consequences folder to the production/consequences folder. Only copy the final models; do not copy the archived models.*

6.  *Copy the contents of the conformance/seeds folder to the production/seeds folder.*

7.  *Using the event generator manifest from conformance, update the blocks per realization to target 2,000 blocks per realization and update the target total events to be 1 million.*

8.  *Run event one starting with storm selection and computing through consequences to verify all linkages continue to function properly in production.*

9.  *Run realization one through HEC-HMS. Verify none of the hydrologic model simulations in the realization failed. Review computed frequency curve for realization one to ensure validity.*

10. *For any failed events, if the failed events require changes to the hydrologic model, create a hot fix folder in the production/hydrology folder with the hot fix model with the date of upload included in the name (e.g., duwamish20250115). Update HEC-HMS manifest to use hot fix models. Recompute failed events using hot fix models.*

11. *Run realization one through system response. Verify none of the system response simulations in the realization failed. Review computed frequency curve for realization one to ensure validity.*

12. *For any failed events, if the failed events require changes to the system response curves, create a hot fix folder in the production/system-response folder with the hot fix model with the date of upload included in the name (e.g., duwamish20250115). Update system response manifest to use hot fix models. Recompute failed events using hot fix models.*

13. *Run realization one through HEC-ResSim. Verify none of the reservoir operations model simulations in the realization failed. Review computed frequency curve for realization one to ensure validity.*

14. *For any failed events, if the failed events require changes to the reservoir operations model, create a hot fix folder in the production/reservoir-operations folder with the hot fix model with the date of upload included in the name (e.g., duwamish20250115). Update HEC-ResSim manifest to use hot fix models. Recompute failed events using hot fix models.*

15. *Using the event level maximum files from the hydrology, system response, and reservoir operations, produce a list of events describing which hydraulics models to simulate for realization one.*

16. *Run realization one through HEC-RAS using the specified list of events. Verify none of the hydraulic model simulations in the realization failed. Review computed frequency curve for realization one to ensure validity.*

17. *For any failed events, if the failed events require changes to a hydraulic model, create a hot fix folder in the production/hydraulics folder with the hot fix model with the date of upload included in the name (e.g., duwamish20250115). Update HEC-RAS manifest to use hot fix models. Recompute failed events using hot fix models.*

18. *Run realization one through Consequences. Verify none of the consequences model simulations in the realization failed. Review computed frequency curve for realization one to ensure validity.*

19. *For any failed events, if the failed events require changes to the consequences model, create a hot fix folder in the production/consequences folder with the hot fix model with the date of upload included in the name (e.g., duwamish20250115). Update Consequences manifest to use hot fix models. Recompute failed events using hot fix models.*

20. *Run realizations 2-50 through HEC-ResSim. Verify none of the hydrology, system response, and reservoir operations events failed.*

21. *For any failed events, if the failed events require changes to a model, create an appropriate hot fix folder in the appropriate model directory within the production folder with the hot fix model with the date of upload included in the name (e.g., duwamish20250115). Update the appropriate manifests to use hot fix models. Recompute failed events using hot fix models.*

22. *Using the event level maximum files from the hydrology, system response, and reservoir operations, produce a list of events describing which hydraulics models to simulate for realization 2-50.*

23. *Run realizations 2-50 through Consequences. Verify none of the hydraulics simulations, resulting mapped outputs fail, or consequences models failed.*

24. *For any failed events, if the failed events require changes to a model, create an appropriate hot fix folder in the appropriate model directory within the production folder with the hot fix model with the date of upload included in the name (e.g., duwamish20250115). Update the appropriate manifests to use hot fix models. Recompute failed events using hot fix models.*

25. *Create summary outputs including mean and standard deviation AEP and recurrence interval grids for depth, velocity, and duration.*
